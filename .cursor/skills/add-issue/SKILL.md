# 登记 issue（docs/issues/）

把一个"暂不修复"的问题落成 `docs/issues/<NNN>-<slug>/README.md`（一个 issue 一个文件夹）。**仅用于暂缓处理的问题**——马上修的直接改代码、不登记。

## 前置：先读管理规则

动手前先读 `docs/issues/README.md`，确认命名、格式、字段约定没有变化（README 是 single source of truth，本 skill 只是操作流程）。

## 步骤

1. **定编号**：查 `docs/issues/` 下已有 `<NNN>-*/` 目录，取最大编号 +1（3 位、连续递增；首个为 `001`）。编号只在本目录内排，与 `docs/requirements/` 无关。
2. **拟 slug**：kebab-case 英文短语，一眼看出问题主题（如 `memory-weak-time-sense`）。
3. **收集信息**（缺哪项就问用户或从上下文补全）：
   - 一句话标题
   - 现象（实际看到什么；能复现就附复现步骤 / 日志 / 终端片段）
   - 根因初判（定位不了就写"待定位"+ 怀疑点）
   - 影响与判断（影响范围、是否违反某需求 AC、为什么暂不修）
   - 可选修复方向（想到的路子，给未来留线索）
   - 严重度：`low`（体感/技术债）/ `medium`（明显但可规避）/ `high`（核心体验或数据正确性）
   - 关联需求：指向 `docs/requirements/<NNN>-xxx/`，无则填"无"
4. **写文件** `docs/issues/<NNN>-<slug>/README.md`，套用下方模板。若有复现日志 / 截图 / 脚本等附带资料，放进同目录（如 `assets/`）。
5. **回报用户**：新文件路径 + 编号 + 一句话标题。不擅自修代码。

## 文件模板

```markdown
# <NNN> · <一句话标题>

- **状态**：open
- **严重度**：<low | medium | high>
- **发现日期**：<YYYY-MM-DD>
- **关联需求**：<NNN-xxx 或 无>

## 现象
<实际看到什么；尽量附复现步骤 / 日志 / 终端片段>

## 根因初判
<目前判断的成因；未定位则写"待定位"并列怀疑点>

## 影响与判断
<影响范围、是否违反某需求 AC、为什么暂不修>

## 可选修复方向
<已想到的几条路子，不强求落地>
```

## 注意

- 日期用今天的真实日期（按用户环境时区）。
- 不删旧 issue；本 skill 只负责新增。关闭 issue 是把状态改 `resolved` + 文末补修复指向，不在本 skill 范围。
- 登记纯属文档动作，**不触发任何代码改动**。
