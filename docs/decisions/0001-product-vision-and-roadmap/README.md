## 1. 产品愿景（Vision）

### 1.1 一句话愿景

> **做一个真正像"朋友"的虚拟陪伴体——而不是一个长得像朋友的工具。**

### 1.2 本质与差异化

- **本质**：通过文字（以及未来的语音）与 LLM 交流。
- **差异化**：在 LLM 之上叠加一整套"像真人"的能力层（记忆、形态、感官通道、人格），让用户在长期使用后，感觉到的不是"一个工具"，而是"一个朋友"。

### 1.3 三条不可妥协的核心体验原则

这三条贯穿项目所有阶段，任何模块的设计若与之冲突，应优先服从这些原则：

1. **像真人，不像工具**
   所有可见的交互、话术、错误反馈，都应尽量贴近"一个朋友会怎么说话"，而不是"一个产品会怎么提示用户"。
2. **记忆是第一护城河**
   记忆系统的健壮性 > 任何花哨的形态。终极目标是让用户**几乎察觉不到记忆的边界**——它不是一个"会失忆的 agent"，不是会说"我不记得这个了 / 作为 AI 我没有持续记忆"的工具，而是一个真的记得你的人。
3. **底层可替换，上层稳定**
   LLM、存储、ASR/TTS 等底层依赖应当是可替换的；上层暴露给"形态层"（桌宠/IM/语音）的接口应当稳定。

## 2. 终极形态描述

### 2.1 用户视角

理想状态下，用户会这样感知它：

- 桌面上常驻一个**小形象**（桌宠形态），可以一直陪着自己。
- 想说话的时候**直接对它讲**，它能听见（麦克风），也能用**文字或语音**回应。
- 不在电脑前的时候，可以通过**常用的 IM 平台**继续和它聊。
- 它**真的记得**自己——记得名字、记得喜好、记得上周聊过什么、记得自己养的猫叫什么——而不是每次都像第一次见面。
- 它有自己**相对稳定的性格**，不是每次回答都像换了个人格。

### 2.2 系统视角

为了支撑上面的体验，系统至少需要这些能力模块（详见第 3 节）：

```
                  [ 形态层 ]
            桌宠 / IM 接入 / CLI
                    │
                    ▼
         [ 感官通道层 ]
       文字 │ 语音输入(ASR) │ 语音输出(TTS)
                    │
                    ▼
              [ 核心能力层 ]
         对话引擎 │ 记忆系统 │ 人格/情绪
                    │
                    ▼
              [ 基础设施层 ]
        LLM Provider │ 存储 │ 跨平台运行时
```

## 3. 能力模块清单

每个模块的"对应需求"列指向 `docs/requirements/` 下已立项或拟立项的需求文件夹。

| 模块                  | 定位             | 状态         | 对应需求                                    |
| --------------------- | ---------------- | ------------ | ------------------------------------------- |
| M1 · 基础对话         | 核心能力         | 孵化期实现   | `001-foundation-chat-and-memory`            |
| M2 · 基础记忆         | 核心能力         | 孵化期实现   | `001-foundation-chat-and-memory`            |
| M3 · 调试 UI          | 开发自测         | 孵化期实现   | `001-foundation-chat-and-memory`            |
| M4 · 桌宠形态         | 形态层           | 待立项       | （未来）                                    |
| M5 · 跨平台支持       | 基础设施层       | 待立项       | （未来，与 M4 强耦合）                      |
| M6 · 语音输入（ASR）  | 感官通道层       | 待立项       | （未来）                                    |
| M7 · 语音输出（TTS）  | 感官通道层       | 待立项       | （未来）                                    |
| M8 · IM 通道接入      | 形态层           | 待立项       | （未来）                                    |
| M9 · 强记忆系统升级   | 核心能力（升级） | 待立项       | （未来，是 M2 的延续，不是替换）            |
| M10 · 人格与情绪      | 核心能力         | 待评估       | （未来；属于"像真人"的进一步延伸）          |

### 3.1 各模块要点速记

- **M1 基础对话**：能多轮、能管理上下文、底层 Provider 可替换。详见 `001/requirement.md`。
- **M2 基础记忆**：能持久化、跨会话不丢、能召回。本期只追求"够用"，不追求"完美"。详见 `001/requirement.md`。
- **M3 调试 UI**：CLI / 最简 Web，**不投入设计资源**。详见 `001/requirement.md`。
- **M4 桌宠形态**：常驻桌面、合理形象、与系统其他能力解耦。
- **M5 跨平台支持**：明确目标平台为 **Windows + macOS**。技术栈选择需考虑双端实现成本。
- **M6 语音输入（ASR）**：通过麦克风采集 → 转文字 → 接入对话引擎。需要解决唤醒/断句/隐私等问题。
- **M7 语音输出（TTS）**：让陪伴感更强；音色应符合人格设定。
- **M8 IM 通道接入**：可挂接一个或多个 IM 平台，用户在 IM 中发文字也能对话。具体平台清单待评估。
- **M9 强记忆系统升级**：长期/短期记忆分层、向量检索、遗忘机制、自我反思总结。终极目标是让"失忆瞬间"不再出现。
- **M10 人格与情绪**：相对稳定的人格特质 + 会因事件波动的情绪状态。这是"像真人"的延伸推断，是否纳入正式路线图待评估（见第 6 节）。

## 4. 路线图（按阶段，不绑日期）

> 本路线图**不承诺时间点**，只承诺**顺序与依赖关系**。每个阶段的进入门槛是上一阶段的核心模块达到验收标准。

### Phase 0 · 孵化期（当前）

- 目标：把地基跑通——能聊、能记。
- 范围：M1 + M2 + M3
- 退出条件：满足 `001/requirement.md` 的 AC-1 ~ AC-5 验收标准。

### Phase 1 · 形态期

- 目标：让"AI 朋友"以一个**可被看见的实体**出现在桌面上。
- 范围：M4（桌宠形态）+ M5（跨平台支持）
- 关键挑战：技术栈选型必须能同时撑住 Windows 与 macOS，且能承载一个常驻、可交互的桌面形象。

### Phase 2 · 感官期

- 目标：让对话不再局限于"打字"。
- 范围：M6（语音输入）+ M7（语音输出）
- 关键挑战：唤醒方式、隐私、延迟、音色。

### Phase 3 · 通道期

- 目标：脱离"必须坐在电脑前"的限制。
- 范围：M8（IM 通道接入）
- 关键挑战：IM 平台选型、消息收发的可靠性、跨通道身份/记忆的一致性（IM 上的"它"和桌面上的"它"必须是同一个）。

### Phase 4 · 拟人期

- 目标：从"功能上像朋友"升级为"心智上像朋友"。
- 范围：M9（强记忆系统升级）+ M10（人格与情绪，若评估通过）
- 关键挑战：记忆抽取/检索/遗忘策略、人格的稳定性与情绪的合理性。

### 阶段顺序的可调性

> 上述顺序是**当前的建议顺序**，并非铁律。例如：若 M9（强记忆升级）在 Phase 0 末期就显得迫切，可以提前；若 IM 接入有外部机会窗口（例如某 IM 平台开放新接口），可以插队到 Phase 1 之前。
> 但**任何提前/插队都应在本文档中显式记录**，避免顺序变成口头共识。

**顺序调整记录：**

- **2026-06-10 · Phase 1 形态层提前启动**：为做正经前端（其本身即未来桌宠的前端层），Phase 1（形态期）提前启动，但本轮只交付最低成本的 L2 收窄版骨架，非完整 Phase 1。详见 [`0003-frontend-stack-and-phase1-kickoff`](../0003-frontend-stack-and-phase1-kickoff/README.md)。

## 5. 核心体验原则的可观测形式

为了让"像真人"不沦为一句口号，每个模块在落地时都必须能给出至少一项**可观测的体验底线**。例如：

| 原则                 | 对应可观测底线                                                                |
| -------------------- | ----------------------------------------------------------------------------- |
| 像真人，不像工具     | 错误反馈应使用拟人话术，禁止暴露技术错误码                                    |
| 记忆是第一护城河     | 在记忆中存在相关信息时，禁止出现"我不记得 / 我没有持续记忆"类话术              |
| 底层可替换，上层稳定 | LLM Provider、存储等底层依赖切换时，不应改动上层模块的代码                    |

未来每个需求文档（`requirement.md`）都应在自己的"验收标准"中体现至少一条本表中的底线。

## 6. 开放问题 / 待评估

以下问题不在本文档中拍板，会在后续单独的决策文档或对应需求文档中讨论：

- **OQ-1 M10（人格与情绪）是否纳入正式路线图？**
  当前是"延伸推断"，需要进一步评估其投入产出比，以及是否会带来不可控的副作用（例如"AI 闹脾气不回话"会不会反而损害陪伴感）。
- **OQ-2 桌宠形态的形象方向？**
  抽象图形 / 拟人 / 拟物（动物、玩偶）？将影响 M4 的实现成本与情感投射效果。
- **OQ-3 IM 通道接入哪几家？**
  涉及合规、接口稳定性、目标用户分布等，需要单独评估。
- **OQ-4 是否需要"多用户/多人格"？**
  当前隐含假设是"一个用户拥有一个 AI 朋友"，是否需要支持一个用户拥有多个不同人格的 AI 朋友？

## 7. 重新评估的触发条件

本文档应在出现以下情形之一时被重新审视并可能更新：

- 任意阶段（Phase）的退出条件被达成
- 出现新的核心能力模块需要纳入清单
- 路线图顺序需要调整
- 三条核心体验原则的任意一条出现争议
- 关键开放问题（OQ-x）有了结论

每次重大修订应保留历史记录（通过 git 历史即可），不需要为每次修订单开新文件。
