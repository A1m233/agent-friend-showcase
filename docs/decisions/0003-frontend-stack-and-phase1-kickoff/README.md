## 1. 背景与决策范围

### 1.1 背景

0002 §3.7 当初把"前端框架（React / Vue / Svelte）"与"锁版本"都推迟到 Phase 1 启动时再定；§3.6 也把 Tauri 透明窗口的真实表现验证留到 Phase 1 spike。

现在我们要做的不是调试 UI，而是一个**正经面向使用的前端**（详见 `docs/explorations/initial-frontend/`）。由于 Tauri 应用 = Web 前端 + Rust 壳，这次用 Web 栈做的对话界面**本身就是未来桌宠的前端层**，演进是叠加而非替换。因此：

- 前端框架这个待定项被**迫提前锁定**；
- 这一动作实质上是**提前启动了 0001 路线图里的 Phase 1（形态期）**——尽管本轮只做其中最低成本的一小块骨架。

### 1.2 决策范围（In Scope）

- 前端框架选型（React）
- CSS 技术选型（Tailwind CSS）
- 配套构建工具基线
- 对 0001 路线图"Phase 1 提前启动"的显式记录

### 1.3 不在范围（Out of Scope，归对应需求 `design.md`）

以下虽与前端相关，但属于**单需求实现层**，由前端需求（`docs/requirements/010-*`）的 `design.md` 决定，本决策不锁：

## 3. 各项详述

### 3.1 前端框架：React

- **候选**：React / Vue / Svelte
- **决策**：React
- **理由**：
  - 0002 §3.7 已明确框架"按开发效率（最顺手）选即可，不被桌宠绑死"——Live2D 走 PixiJS（canvas/WebGL）框架无关，框架选择不受形态层约束。
  - 生态池子最大：自写数据流、AG-UI 官方 TS 客户端、未来可能用到的对话/可视化组件，React 选择面最广。
  - 开发者前端背景下的主流稳定选择，长期招人 / 协作成本低。
- **放弃 Vue 的理由**：可用，但部分 AI 前端生态（如某些 AG-UI 周边、headless 组件）以 React 为母语；在"框架无强制约束、按顺手选"的前提下 React 池子更大。
- **放弃 Svelte 的理由**：生态规模与团队熟悉度不及 React，在本项目无差异化收益。
- **重评触发**：React 出现重大方向性问题，或团队整体转向其他框架。

### 3.2 构建工具：Vite

- **决策**：Vite
- **理由**：Tauri 官方模板首选、与 React + TS 组合开箱即用、dev server 与多入口（单 build 多 entry，对应桌宠窗 / 对话窗）支持成熟。
- **重评触发**：出现明显更优且与 Tauri 集成更顺的构建方案。

### 3.3 CSS 技术：Tailwind CSS

- **候选**：Tailwind CSS / CSS-in-JS（styled-components 等）/ CSS Modules / 全量组件库自带样式体系
- **决策**：Tailwind CSS（配合 CSS 变量做主题/换肤）
- **理由**：
  - 与本项目个人 rule「在不破坏风格与架构前提下尽量用 tailwindcss」一致。
  - 与"CSS 变量换肤"方案天然契合：主题切换通过 `html[theme='xxx']` 注入同名不同值的 CSS 变量，Tailwind token 映射到这些变量，切主题即整体换肤。
  - 与后续选用的组件方案（基于 Tailwind + CSS 变量的 copy-in 组件）同一套体系，不引入第二套样式/主题系统。
- **放弃 CSS-in-JS 的理由**：运行时开销 + 与 CSS 变量换肤方案叠床架屋。
- **放弃全量组件库自带样式体系的理由**：会与 CSS 变量换肤形成两套主题系统，维护成本高（详见对应前端需求 `design.md` 的组件方案取舍）。
- **边界**：本决策只锁"CSS 技术 = Tailwind"。具体的主题/换肤实现、组件库/组件方案选型属单需求实现层，归对应前端需求的 `design.md`（见 §1.3）。
- **重评触发**：Tailwind 出现重大破坏性变更，或团队整体转向其他样式方案。

### 3.4 前端版本基线

- **决策**：沿用并锁定 0002 §3.7 的方向 —— Node LTS（22+）+ TypeScript 5.x + pnpm + Tauri 2.x。
- **说明**：0002 §3.7 当初状态为"先记方向，Phase 1 启动时锁版本"；本决策即为该锁定动作。具体 patch 版本随脚手架初始化时落定，不在本文档逐位锁死。

### 3.5 Phase 1 形态层提前启动

- **决策**：本轮起，0001 路线图的 **Phase 1（形态期）提前启动**。
- **范围克制**：本轮**不是**实现整个 Phase 1，只交付**最低成本的形态层骨架**（exploration 中的"L2 收窄版"）：
  - Tauri 透明悬浮窗作桌宠主窗 + **占位形象**（预留未来 Live2D 的 PixiJS 挂载点）；
  - 窗内操作栏浮层（本轮仅一个按钮：打开常规对话界面）；
  - 传统 chatbot 对话界面，接 `agent-bridge` 跑通对话（本轮核心价值）。
- **明确不做（留口子）**：Live2D 实装、情绪/状态事件通道、复杂动作表情、跨平台打包签名等完整 Phase 1 能力。
- **理由**：做正经前端必然落在形态层；与其在 Phase 0 之外含糊推进，不如显式承认"Phase 1 提前并收窄"，符合 0001 §4 对"提前/插队需显式记录"的要求。
- **重评触发**：L2 骨架验证完成后，评估是否继续推进 Phase 1 剩余部分（Live2D、情绪通道、跨平台打包等）。
