# 决策 0005 · 语音交互形态

> Voice Interaction Form
>
> 本文档锁定 `agent-friend` **语音交互的产品形态**这一项目级长期姿态——语音必须由用户**显式触发**，不做 always-on 监听；以及 V1 路线（显式拨号）+ 公网穿透在产品语义中的位置。
> 本文档只决"语音以什么形态在场"——具体到前端怎么集成 RTC SDK、voice_bridge 怎么进程托管、通话浮层 UI 怎么画、wake-word 引擎怎么选，由后续需求的 `requirement.md` / `design.md` 或前置 spike 决定，不在本文档范围内。

## 1. 背景与决策范围

### 1.1 背景

[需求 007](../../requirements/007-voice-call/requirement.md) 已经把 voice_bridge 控制平面 + LLM 入站代理 + agent 层 `channel_change` 机制全部合入，且 2026-06-15 在 Win 上跑通了真火山 RTC 链路的 smoke。技术底座已经在位。

但准备把语音真正接入桌面端（029 系列需求即将立项）时，浮出一个比"前端集成怎么做"更上层的问题——**语音交互在用户面前长什么样**：

- 是"打电话 / 挂电话"这种显式动作？
- 还是"agent 一直能听到你说什么"这种环境式存在？
- 还是按住说话的 push-to-talk？
- 还是 Siri 那种本地唤醒词触发？

这个问题不是某一个需求的局部决策——它跨所有当前与未来的 voice surface（pet 形态接 voice / chat 形态接 voice / 未来 IM 通道接 voice / 客户端形态接 voice），必须在 029 立项前作为项目级姿态锁定，否则每接入一个 surface 都要重新讨论一遍，且很容易在不同 surface 之间长出语义冲突。

### 1.2 决策范围（In Scope）

- 语音监听模式的项目级长期姿态（显式触发 vs 环境监听）
- 各候选形态（always-on RTC / push-to-talk / 显式拨号 / wake-word）的当下取舍
- V1 路线锁定
- 公网穿透在产品语义中的位置（用户显式接受 vs 隐藏 / 自动化）
- "为什么不做"的显式登记
- 路线反转触发条件

### 1.3 不在范围（Out of Scope，归对应需求 / spike）

- **桌面端前端怎么集成 `@volcengine/rtc`、通话浮层 UI / ActionBar 拨号按钮形态** → 后续桌面端语音接入需求 `requirement.md` / `design.md`
- **voice_bridge / cloudflared 的 Tauri sidecar 进程托管 / 动态公网 URL 注入** → 后续后端进程托管需求 `requirement.md` / `design.md`
- **V2 候选形态的具体引擎 / 方案选型**（如假设走 wake-word，是 Picovoice / Sherpa / 火山 / 别的）→ V1 合入后的前置 spike
- **语音通话相关 settings UI**（凭证录入、公网穿透开关、通话权限）→ 未来基于 028 设置中心扩展的需求
- **延迟优化**（007 progress 已登记的"用户说完 → AI 开始回答 ≈ 8s"）→ 单独立项
- **隐私文案 / 法务合规视角的最终用语** → 合规 review 流程承接，不在本 ADR 展开
- **凭证管理 / 多租户 / 鉴权 / 速率限制** → 沿 [007 §3](../../requirements/007-voice-call/requirement.md) 同精神不做

## 2. 决策

### 2.1 决策清单速查表

| # | 项 | 决策 |
|---|---|---|
| 1 | 语音监听模式（项目级长期姿态） | **必须显式触发**；所有 voice surface 都遵循此姿态 |
| 2 | always-on RTC 持续监听 | **明确不做**（成本 + 隐私 + 心智模型三重否定，详见 §3.1） |
| 3 | V1 形态 | **显式拨号**（用户在 surface 上发起通话 → 进入 RTC session → 挂断释放） |
| 4 | V2 形态 | **本 ADR 不锁定**；候选方向由 V1 合入后的前置 spike 评估后另立 ADR 或在本 ADR 上延伸 |
| 5 | push-to-talk | **短期内不考虑**（与"陪伴体在场"vision 冲突，但不作为永久否决） |
| 6 | 公网穿透在产品中的位置 | **用户显式接受**：开启语音通话 = 接受功能依赖公网穿透；UI 上明示，不偷偷拉穿透 |
| 7 | 通话期间的 channel 切换 | 沿 007 已实现的 `channel_change` 机制：发起通话 = channel 升 voice，挂断 = 降回 text；不在本 ADR 重复 |

### 2.2 产品语义

语音 = **用户主动发起的一段对话窗口**，不是"agent 持续在场监听环境"。

形态上的体感差异：

- **显式触发**：用户清楚知道"我现在在跟它说话"，跟"现在不是在跟它说话"两个状态有明确边界；麦克风占用指示符（OS 顶栏 / 任务栏的橘点 / 红点）只在用户主动开启时亮起
- **环境监听**：麦克风指示符 24/7 长亮，用户无法分辨"它在听 vs 它没在听"，与日常使用的所有 app 都形成认知冲突

这跟 [`0001`](../0001-product-vision-and-roadmap/README.md) §1.1 "做一个真正像朋友的虚拟陪伴体——而不是一个长得像朋友的工具" 的诚实性原则一致：**朋友会让你知道它什么时候在听你说话**。

### 2.3 V1 / V2 关系

- **V1**（显式拨号）锁定后即可启动 029 系列接入需求，路径清晰，不阻塞
- **V2**（无论最终走 wake-word 还是别的）目标是降低"显式"的成本——让用户更自然地起话，但**必须仍然在"显式触发"的姿态之下**；环境监听不会因为"V2 来了"而被翻案
- V2 候选方向由前置 spike 评估后决定，可能的方向（不构成承诺）包括：本地 wake-word 唤醒词、快捷键热键、声纹/能量阈值检测启动等；具体看 spike 结论

## 3. 决策依据

### 3.1 为什么 always-on RTC 监听明确不做

三重否定，任何一条都足以否决，叠加更不可逆。

**经济性**：火山 RTC AIGC 按通话分钟 + ASR/TTS 字符 + AIGC 任务时长计费，且火山 AIGC 任务有最大时长上限（spike 期 `Custom.json` 配置可见），24h 常驻一台机一天数十到数百人民币，技术上拉不起、商业上不可持续。即便未来出现端侧 ASR/TTS 方案，VAD（语音活动检测）+ 模型推理本身仍非零成本。

**隐私 / 心智模型**：

- 操作系统的麦克风占用指示符（macOS 顶栏橘点、Windows 任务栏录音图标）会 24/7 长亮，用户每分钟看到，体感等同窃听软件
- 用户无法分辨"它在听 / 它没在听"两个状态，与日常使用的所有 app（按需启用麦克风）形成认知冲突
- 即便从工程角度可以承诺"音频本地丢弃 / 不上云"，用户视角无法验证此承诺——OS 看到的就是"麦克风在被使用"

**上下文污染**：always-on 会让 agent 听到用户跟家人 / 同事的对话、电话另一头、电视背景音、孩子的哭声。LLM 把这些都当 context，可能：

- 在用户跟别人讲话时主动插话
- 把电话对面或电视里的话当成用户的指令
- 通过记忆系统把"窃听到"的内容写入长期记忆，污染未来所有决策——一旦写入就再难清理

这跟 [`0001`](../0001-product-vision-and-roadmap/README.md) "记忆是第一护城河" 是反向冲突的：第一护城河被环境噪音污染，反成最大风险面。

### 3.2 V1 走显式拨号的依据

- **链路最短**：007 已经在 smoke 流验证过的拨号 → joinRoom → 通话 → 挂断的完整闭环，桌面端只需把 smoke HTML 的逻辑转成产品级 UI，技术风险已知
- **控制平面完整**：发起通话 = 进入一段持续会话，挂断 = 释放火山 RTC 资源 + 关闭麦克风采集 + channel 降回 text，每一步都有清晰的释放语义；崩溃恢复也好处理
- **跟用户固有心智一致**：拨号 / 挂断是 50 年的通信交互范式，用户不需要学习
- **跟 chat 同 session 续接的语义清晰**：voice_bridge `POST /voice/calls` 已支持传入 `session_id` 续上现有 chat 会话，channel 升降级走 007 的 `channel_change`——没有额外协议负担

### 3.3 公网穿透"用户显式接受"的依据

火山 RTC `CustomLLM` 模式协议层硬约束：火山云端必须能反向调用 voice_bridge 的 LLM proxy URL。这意味着 voice_bridge 的 LLM proxy 路径**必须有公网可达入口**——不存在"完全本机闭环"的方案，除非：

- (a) 换不要求公网回调的 RTC vendor（重大架构调整，远期）
- (b) 把 voice_bridge 托管到服务端（违背孵化期"本机优先"原则，远期）

短期内 (a)(b) 都不做，因此公网穿透是用户必须接受的客观事实。两种处理姿态：

- **偷偷起 cloudflared**：用户不知情；某天 quick tunnel 失败 / 被墙 / cloudflared 升级断流时，"语音通话莫名其妙不能用"会让用户彻底丢失对产品的信任
- **显式同意**：开启语音通话功能时，UI 上明示"语音通话需要公网穿透到火山云，是否启用"；用户接受后才拉穿透，状态可见、可以关

后者更诚实，跟 [`0001`](../0001-product-vision-and-roadmap/README.md) §1 vision 的诚实性原则一致。这条姿态对未来的 voice_bridge sidecar 化（穿透自动管理）同样适用——自动化的是"启动后保持运行"，**用户的显式同意环节不能因此消失**。

### 3.4 V2 不锁方向的依据

ADR 一旦锁定方向，未来 spike 即便给出反向结论也要走"新建编号更大的决策推翻"的重流程。V2 当前处于：

- 候选方向有多个（wake-word / 热键 / 别的），还没做对比评估
- wake-word 即便是候选最大宗，引擎选型本身也有风险（商用 license / 开源模型质量 / 跨平台支持 / 自定义唤醒词训练数据）
- V1 还没接入桌面端，用户反馈尚未到达，无法判断"显式拨号的痛点究竟在哪"——V2 应该解决具体痛点而非提前优化

因此本 ADR 只锁**项目级长期姿态**（显式触发）+ **V1 路线**，把 V2 留给"V1 合入 → 用户反馈 → 前置 spike → 候选方案对比"的链路自然决出。这是"晚锁定"原则的应用。

### 3.5 push-to-talk 短期不考虑、但不永久否决的依据

PTT 跟显式拨号同属"显式触发"姿态，逻辑上是兼容的。当前选 V1 = 拨号而非 PTT 的理由：

- PTT 的"按住说话"心智偏工具向（对讲机），跟 0001 "陪伴体在场" vision 张力较强
- PTT 在桌面端的快捷键冲突 / 全局热键权限 / 焦点窗口劫持等工程问题不轻
- 拨号 = 持续会话语义，跟 agent 的"记得上下文 / 持续对话"特性更契合

但不作为永久否决，理由：

- 若 V1 合入后用户反馈"每次都要点拨号 + 挂断太重"，PTT 是一条轻量的降级方案
- 若 V2 方向评估发现 wake-word / 热键等候选都不可行，PTT 可以作为兜底

因此 ADR 登记为"短期内不考虑"，重评条件见 §7。

## 4. 影响范围

### 4.1 对已合入代码的影响

无。007 已经把 channel 切换 / voice_bridge 控制平面 / LLM 入站代理全部落地，本 ADR 不改任何已有代码。

### 4.2 对未来需求的约束

| 需求 / surface | 约束 |
| --- | --- |
| 桌面端语音接入（029 系列待立） | 必须沿"显式触发 + 公网穿透显式接受"的语义设计 UI；不允许出现"自动开始监听 / 用户不知情拉穿透"的交互 |
| 未来 IM 通道接 voice（022 后续扩展） | 同样遵循显式触发——IM 侧拨号语义可适配 IM 平台原生（如"语音通话邀请"），但不允许 always-on |
| 未来桌宠 / pet 形态语音 | 同上；pet ActionBar 上的拨号按钮是显式触发的一种实现 |
| 客户端"操控电脑"形态语音命令 | 同上；不允许用"agent 听 OS 全局声音"的方式触发命令 |

### 4.3 对 settings / 隐私文案的约束

未来 settings 中"语音通话"分类必须：

- 默认关闭（用户主动开启才生效）
- 开启时同时呈现"功能依赖公网穿透"的同意项
- 提供随时关闭的入口
- 文案合规 review 不在本 ADR 范围

## 5. 接受的 trade-off

显式登记本 ADR 接受的代价，避免未来反复重提。

- **T-1**：**用户必须接受公网穿透才能用语音**——拒绝 always-on 的代价之一是把"必须公网回调"这个协议层约束暴露给用户。后续可演进（服务端托管 / 换 vendor）但本 ADR 不锁定演进路径。
- **T-2**：**V1 体感不如 always-on 自然**——"agent 直接听到你说话"的环境式存在感确实更接近某些科幻作品的"AI 伙伴"想象。本 ADR 选择牺牲这种体感，换取隐私边界清晰 + 成本可控 + 用户心智明确。这是有意识的产品姿态选择，不是技术受限的妥协。
- **T-3**：**V2 方向留白带来不确定性**——下游需求 / 用户期待 V2 的具体时间表无法从 ADR 直接读出。补偿：V1 合入即启动前置 spike，候选方案对比 + 选型决定后另立 ADR 或在本 ADR 延伸。
- **T-4**：**PTT 留口子带来轻微"半决策"风险**——未来若用户反馈强烈，"短期不考虑"会被重提。补偿：§7 写明重评触发条件，避免凭感觉拉扯。

## 7. 重新评估触发条件

满足任一即应启动 ADR 重评（可能新立 0006+ 推翻本 ADR 的某项）：

- **always-on 否决相关**
  - 出现支持端侧 ASR/TTS + 端侧 LLM 推理且性能 / 隐私都可接受的方案，"环境监听"的隐私和成本论点都失效
  - OS 层出现"应用可声明背景麦克风访问且不亮全局指示符"的机制（极不可能但留口）
- **公网穿透相关**
  - 出现不要求公网反向回调的 RTC vendor，且接入成本可接受
  - voice_bridge 服务端托管的远期方案落地，公网穿透变成服务侧细节
- **V2 留白相关**
  - V2 候选方向评估 spike 全部失败 → 需重新评估是否仅靠 V1 长期承担
  - V2 spike 给出明确方向 → 本 ADR §2.1 #4 延伸 / 另立 ADR
- **PTT 相关**
  - V1 桌面端接入后用户群体明确反馈"每次拨号 + 挂断太重"，且 wake-word / 热键候选都未落地
- **vision / roadmap 相关**
  - [`0001`](../0001-product-vision-and-roadmap/README.md) 重大方向变更，例如把"陪伴体"姿态改为"工具"姿态——本 ADR 的 PTT / always-on 取舍可能随之重评
