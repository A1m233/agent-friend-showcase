# 决策记录（Decisions）

本目录存放 `agent-friend` 项目级的**重大决策记录**。

## 适用范围

任何**影响范围超出单个需求、且后续修改成本较高**的决策都应在这里登记，包括但不限于：

- 产品愿景与路线图
- 技术栈选型
- 架构原则
- 跨模块的协议/规范
- 重要的"不做"决定（例如"为什么不接入某 IM 平台"）

> 仅服务某个需求的决策不属于这里，应放在该需求的 `design.md`。

---

## 目录约定

每个决策都是一个**独立文件夹**，命名格式为 `<4 位数字编号>-<kebab-case 英文短语>`：

```
docs/decisions/
├── README.md                              ← 本文件，仅维护管理规则
├── 0001-product-vision-and-roadmap/
│   ├── README.md                          ← 决策主文档
│   └── （未来可放图、调研笔记、子决策等）
└── 0002-incubation-tech-stack/
    ├── README.md
    └── ...
```

约定要点：

- **主文档统一叫 `README.md`**：GitHub 访问文件夹时自动渲染，体验最好。
- **编号 4 位数字、连续递增**：方便排序，4 位预留扩展空间。
- **kebab-case 英文短语**：跨平台路径友好。
- **一个决策一个文件夹**：未来可在同一决策下放图、调研笔记、相关参考资料、子决策、历史版本等，避免一个 .md 文件越长越乱。

---

## 修改约定

- **决策一旦发布，不就地推翻**：如果未来要修改方向，应**新建编号更大的决策记录覆盖**（例如 0005 推翻 0002 的某项），保留决策演进历史，而非默默改原文。
- **可以原地修订的情况**：仅限于补充细节、修正笔误、更新链接、更新元信息状态等不改变决策本意的内容。
- **可以更新的元信息**：状态（草稿 → 生效 → 被 0005 覆盖）、重评触发条件等。

---

## 与其他文档的边界

| 文档类型           | 位置                                    | 职责                                       |
| ------------------ | --------------------------------------- | ------------------------------------------ |
| **决策（本目录）** | `docs/decisions/`                       | 项目级决策、长期不易变的方向               |
| **需求**           | `docs/requirements/<id>/requirement.md` | 描述某个具体需求要解决什么问题             |
| **技术方案**       | `docs/requirements/<id>/design.md`      | 在已锁定的项目级技术栈下，某个需求怎么实现 |

简单原则：

> **如果一个决策只服务某一个需求 → 写在那个需求的 `design.md` 里。**
>
> **如果一个决策影响项目长期方向、跨多个需求 → 写在这里，新建一个决策文件夹。**
