# 001 · 基础对话与记忆能力

> Foundation: Chat & Memory
>
> 项目孵化期的第一个需求，也是后续一切能力的地基。

---

## 1. 背景与项目定位

### 1.1 项目一句话定义

`agent-friend` 是一个**虚拟朋友**定位的 AI 项目。

它的本质：通过文字与 LLM 交流。
它的差异化：在 LLM 之上叠加一整套"像真人"的能力层（记忆、人格、形态、感官通道等），让用户在长期使用后，感觉到的不是"一个工具"，而是"一个朋友"。

### 1.2 为什么先做"对话 + 记忆"

整个产品会演化为一个庞大的系统（桌宠形态、跨平台、语音、IM 通道、强记忆等）。但所有这些能力的下面，都站着同一个地基：

- **对话能力**：能通过文字与 LLM 进行有意义的多轮交流。
- **记忆能力**：能跨会话、跨时间地记住与用户相关的事实，并在合适的时候召回。

地基跑不通，上面所有形态都是空中楼阁。因此孵化期只做这两件事，加上一个**纯供开发自测使用**的简易界面。

## 4. 核心需求详述

### 4.1 对话模块

**目标**：让"人"和"AI 朋友"之间能进行自然的多轮文字对话。

需求点：

- **R-4.1.1 多轮对话**：用户输入一条消息，AI 给出回复，能基于此前的上下文继续对话。
- **R-4.1.2 上下文管理**：当对话长度超过模型上下文窗口时，应有合理的截断 / 压缩 / 摘要策略，而不是直接报错或丢失。
- **R-4.1.3 LLM Provider 抽象**：底层调用哪一家 LLM 服务，应当是**可替换的**。本期可以只接入一家，但代码结构上不能与某家强耦合。
- **R-4.1.4 错误与重试**：对 LLM 调用失败、超时、限流，需要有兜底处理（至少不能让进程崩溃，要给用户一个可读的反馈）。

> 具体的上下文管理策略、Provider 抽象方式、错误处理机制，由 `design.md`（技术文档）决定。

### 4.2 记忆模块

> **落地承接**：本节记忆能力（及下方 AC-3 / AC-4 / AC-5）的真正实现，已上移为独立需求 [`008 引擎层长期记忆`](../008-engine-memory/requirement.md)。本节保留作为最初的需求来源与背景。

**目标**：让 AI 看起来"真的记得"用户，而不是每次重启都要重新自我介绍。

这是本项目最核心的差异点，也是后续会持续投入的方向。本期只要求"能用、不丢、可召回"，不要求达到最终形态。

需求点：

- **R-4.2.1 写入 / 提取**：需要从对话流中识别出"值得记住的信息"——例如用户的姓名、偏好、重要事件、与 AI 之间发生过的事。本期允许使用最朴素的策略（如全量写入或简单 LLM 抽取）。
- **R-4.2.2 持久化存储**：记忆必须落盘，进程重启后不丢。
- **R-4.2.3 跨会话不丢**：用户在第 1 次对话中告诉 AI "我叫小明"，第 N 次对话开启时，AI 仍应能在合适时机自然地用到这个信息。
- **R-4.2.4 检索召回**：在生成回复前，需要根据当前对话上下文，从记忆中召回相关条目，作为额外上下文喂给 LLM。
- **R-4.2.5 体验底线**：禁止出现**暴露 AI / 模型 / 系统身份**的话术，例如 "作为 AI"、"作为大语言模型"、"我是 GPT / Claude / DeepSeek"、"我没有持续记忆能力"、"我的训练数据截止到…" 等。**忘事本身是正常的**——真人也会忘——只要话术像真人会说的（如 "嗯…我有点记不太清了，你再说说？"），就不违反体验底线。

> 记忆的存储介质（SQLite / 向量库 / 文件 / 混合方案）、抽取策略、召回算法，由 `design.md` 决定。

### 4.3 调试用 UI

**目标**：让开发者自己能跑起来、能联调，仅此而已。

需求点：

- **R-4.3.1 能输入文字**：能通过键盘输入消息发送给 AI。
- **R-4.3.2 能看到回复**：能看到 AI 的回复内容。
- **R-4.3.3 能看到记忆相关信息（开发可见）**：例如本轮召回了哪些记忆、本轮新写入了哪些记忆，便于调试。
- **R-4.3.4 形态不限**：CLI、本地 Web 页面、IDE 内调用脚本，任选其一即可。**不投入任何视觉/交互设计资源。**

## 5. 关键体验原则

以下原则不仅服务本期，也服务于未来所有需求。任何后续模块设计若与之冲突，应优先服从这些原则：

1. **像真人，不像工具**：所有可见的交互、话术、错误反馈，都应尽量贴近"一个朋友会怎么说话"，而不是"一个产品会怎么提示用户"。
2. **记忆是核心护城河**：记忆系统的健壮性 > 任何花哨的形态。宁可形态简陋，不可记忆失忆。
3. **底层可替换，上层稳定**：LLM、记忆存储等底层依赖应当是可替换的；上层暴露给"形态层"（未来桌宠/IM/语音）的接口应当稳定。

## 7. 验收标准

> **记忆相关 AC（AC-3 / AC-4 / AC-5）的落地与细化已迁移到 [`008 引擎层长期记忆`](../008-engine-memory/requirement.md)**；下方保留原始描述作为来源。对话相关 AC（AC-1 / AC-2）仍由本需求负责。

本期需求验收时，需要至少满足以下场景：

- **AC-1 基础对话**：能在调试 UI 中开启一段对话，进行至少 10 轮自然往返，回复合理、不崩溃。
- **AC-2 上下文连贯**：在同一段对话内，AI 能正确引用前几轮提到的内容（例如用户 5 轮前说过的事，AI 在第 6 轮被问起时仍能回应）。
- **AC-3 跨会话记忆**：
  - 用户在一次会话中告诉 AI 一个关键事实（例如："我叫小明，我养了一只猫叫 Tom"）。
  - 关闭进程，重新启动。
  - 在新会话中以自然的方式提起相关话题，AI 能正确召回这些事实，且表达自然。
- **AC-4 不出现"失忆"话术**：在记忆中存在相关信息时，AI 不应出现 "我不记得"、"我没有记忆能力" 类话术。
- **AC-5 调试可观测**：开发者能在调试 UI 中看到本轮召回了哪些记忆 / 写入了哪些记忆。
