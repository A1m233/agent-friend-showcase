# 002 · 引擎层会话管理

> Engine-Level Session Management
>
> 把"会话"从 CLI 的实现细节上移为引擎层的一等公民概念。

---

## 1. 背景

### 1.1 当前现状

[001 需求](../001-foundation-chat-and-memory/requirement.md) 的 M0.3 落地后，"会话"的概念已经隐式存在：

- `agent.Conversation` 在内存里维护 history
- CLI 退出时 dump 一份 JSON 到 `data/sessions/session-{timestamp}.json`
- 启动时通过 `--resume` 加载某个 JSON 文件

但**这一整套会话生命周期管理是写在 CLI 里的**：

- `tools/cli/__main__.py` 的 `save_session()` 拼时间戳文件名
- `build_conversation()` 处理 `--resume` 找最新 / 显式路径
- 没有"会话列表"概念，文件名只是时间戳，肉眼无法识别"哪段对话是哪段"
- 切换 persona / model 都不在能力清单内
- `Conversation.dump()` 只存 history，不存任何会话级元数据

### 1.2 为什么要上移到引擎层

按 [`0001 产品愿景与路线图`](../../decisions/0001-product-vision-and-roadmap/README.md) §1.3 第 3 条 **"底层可替换，上层稳定"**，会话管理是一种**所有形态层都会用到的能力**：

- 当前 CLI 需要
- 未来桌宠前端（M4）需要
- 未来 IM 接入（M8）需要
- 未来 HTTP API 需要

如果每个形态层各自实现 dump/load/list：

- 重复劳动
- 格式可能漂移（一个写 JSON、一个写 SQLite、一个写文本）
- 一致性体验难以保证（同一份会话在不同入口看起来不一样）

→ 必须把"会话"作为**引擎层的一等公民**，对外暴露统一的会话管理 API。

### 1.3 与 001 / 未来 memory 的边界

| 概念 | 定位 | 本期范围 |
|---|---|---|
| Session（会话） | 一段连续对话的容器，有起止、可恢复、可列出 | **本期做** |
| Conversation（运行时执行器） | 把 LLM / persona / 上下文管理串起来跑一轮对话 | 001 已做，本期可能微调以接入 Session |
| Memory（长期记忆） | 跨会话的事实 / 偏好 / 事件抽取与检索 | 001 阶段 2 做，**不在本期** |

**Session ≠ Memory**：Session 是"原始对话的载体"，Memory 是从对话中**抽出**的语义化事实。两者平行、互不依赖。会话存储的语义是 **append-only 事件流**，长期记忆的语义是 **可检索的结构化条目**，二者存储介质可以、也应该不同。

---

## 2. 本期范围（In Scope）

### 2.1 引擎层一等公民

- 把会话作为引擎层（`agent/` 模块）的一等公民概念，提供完整生命周期管理
- 任何调用方（当前 CLI、未来 HTTP API、未来前端）通过同一套接口操作会话

### 2.2 持久化与可恢复

- 会话必须落盘，进程重启不丢
- 写入策略需支持**边对话边落盘**（不能只在退出时写）
- 存储介质不强求与 [`0002 决策`](../../decisions/0002-incubation-tech-stack/README.md) §3.15 的 SQLite 一致 —— 会话流是 append-only 事件流，与"长期记忆"的存储语义不同（具体选型见 `design.md`）

### 2.3 会话内动态切换 persona / model

- 用户在一次会话进行中可以切换 persona
- 用户在一次会话进行中可以切换 model
- 切换前后**历史不丢**，下一轮起使用新值
- 切换动作在会话历史中留有**可追溯标记**，未来形态层可以渲染（如展示"你在 X 时刻把人设从 A 改成了 B"）

### 2.4 会话列表 / 打开 / 删除

- 列出当前用户的所有会话（按最近活跃排序）
- 通过会话标识打开任一会话
- 删除会话

### 2.5 CLI 适配

- CLI 的 `--resume` / 自动保存 / `/reset` 等行为改为调用引擎层 API
- CLI 不再直接实现任何会话生命周期逻辑（不再调用 `Conversation.dump/load`、不再拼时间戳路径）

---

## 3. 非目标（Out of Scope）

以下能力**本期明确不做**：

- **会话操作**：archive / fork（从某条消息分支新会话）/ rename / 复制 / 合并
- **多进程并发写**：本期假设单进程使用，多个并发进程同时改同一会话的行为不在保证内
- **写入中崩溃保护**：不做原子写入（fsync + 临时文件 + rename），允许"崩溃丢最后一条事件"
- **Schema 版本迁移**：允许首版 schema 不向前兼容，未来需要迁移再做
- **自动标题生成**：本期不做 LLM 提炼标题；仅在引擎接口上预留 `title_generator` 注入口
- **HTTP API 实现**：仅在接口设计上保证可包装为 HTTP，不在本期落地 HTTP 层（与 001 design §1.2 一致，HTTP 留到形态期）
- **跨设备同步 / 多用户隔离**：单设备单用户假设
- **会话内消息编辑 / 撤回**：不支持
- **会话搜索**（按内容检索）：不支持，按需求迭代再加

## 4. 核心需求详述

### 4.1 会话生命周期

**目标**：调用方通过引擎层 API 完成所有会话操作，不直接接触底层存储。

需求点：

- **R-4.1.1 创建**：能创建一个新会话，会话获得唯一标识；初始 persona / model 由调用方指定
- **R-4.1.2 打开**：能通过会话标识打开一个已有会话，恢复全部历史与当前状态
- **R-4.1.3 追加事件**：用户消息 / AI 回复 / 切换事件等都通过统一的"append event"路径写入
- **R-4.1.4 列出**：能列出所有会话，至少包含会话标识 / 标题 / 创建时间 / 最近活跃时间
- **R-4.1.5 删除**：能删除一个会话，删除后不可恢复（hard delete）

### 4.2 持久化与恢复

- **R-4.2.1 边对话边落盘**：每条消息生成后必须立即写入存储，不允许"只在退出时整体写"
- **R-4.2.2 进程重启不丢**：进程崩溃 / 正常退出后，重新启动可恢复任意已写入的会话
- **R-4.2.3 退化场景容忍**：写入过程崩溃时，允许丢失最后一条未完成事件，但**不能损坏整个会话**

### 4.3 会话内动态切换 persona / model

- **R-4.3.1 切换 persona 不中断会话**：在会话进行中切换 persona，历史保留，下一轮起使用新 persona 的 system prompt
- **R-4.3.2 切换 model 不中断会话**：同 R-4.3.1，model 维度
- **R-4.3.3 切换事件可追溯**：切换动作本身作为可观测的事件写入会话历史，未来形态层可据此渲染"切换发生在哪条消息之后"
- **R-4.3.4 历史消息归属可见**：每条消息可观测到它是在哪个 persona / model 下生成的

### 4.4 可观测性

- **R-4.4.1 列表性能合理**：列出会话时不需要读取每个会话的全部内容；列出耗时与单个会话历史长度**不相关**
- **R-4.4.2 调试可读**：会话文件应为人类可读格式（`cat` / `grep` / `tail -f` 友好），便于开发调试与肉眼审查
- **R-4.4.3 消息标识**：每条消息有唯一标识，方便未来"引用 / fork"等扩展（本期不引入 `parent_uuid`）

### 4.5 CLI 适配

- **R-4.5.1 行为对齐**：CLI 现有行为（多轮 / 流式 / `--resume` / 自动保存 / `/reset` / 人设切换）不退化
- **R-4.5.2 实现下移**：CLI 不再直接调用 `Conversation.dump/load`、不再拼时间戳文件名，全部走引擎层 API
- **R-4.5.3 新增最小命令**：CLI 提供基础会话切换能力（如"列出会话"、"打开会话"、"切换 persona / model"），具体形态由 `design.md` 决定

## 5. 关键体验原则

引用 [`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 的三条原则，本期重点服从：

1. **底层可替换，上层稳定**：会话存储介质（JSONL / SQLite / 远程对象存储等）应当是可替换的；引擎层暴露给形态层的会话 API 应当稳定，本期 API 在阶段 2 / 阶段 3 必须只向后兼容扩展
2. **像真人，不像工具**：切换 persona / model 不应给用户暴露"现在切换到 `model=deepseek-chat`"这类技术细节，形态层可以选择是否展示（本期 CLI 可以简短提示）
3. **记忆是第一护城河**：本期不直接做记忆，但会话流是未来 memory 抽取的原始数据源 —— 因此会话历史必须**完整、有序、可追溯**（含 persona/model 归属）

## 6. 验收标准

- **AC-1 跨进程会话恢复**：用户在会话中说几句话，正常退出 CLI，重新启动后通过 `--resume <id>`（或恢复最新）能看到完整历史
- **AC-2 异常退出不丢全部**：会话进行中 `kill -9` 进程，重新启动恢复时，已写入的消息完整可见（允许丢失最后一条正在写的）
- **AC-3 切 persona 不中断**：进行 5 轮对话后切换 persona，再进行 5 轮，能看到所有 10 轮历史，且第 6 轮起 AI 表现符合新 persona
- **AC-4 切 model 不中断**：同 AC-3 的 model 维度
- **AC-5 切换可追溯**：从会话历史中能定位到 "persona / model 切换发生在哪条消息之后"
- **AC-6 列表不读全文件**：会话数量为 100、单个会话历史长度为 1000 条 vs 10 条时，列出会话耗时**与历史长度无关**（即不为列出而读取全部消息）
- **AC-7 CLI 行为不退化**：001 M0.3 完成的 CLI 行为（多轮 / 流式 / 持久化 / 人设切换 / 错误处理）在新引擎接口下全部继续可用
- **AC-8 持久化为人类可读**：开发者能用 `cat` / `tail` / `grep` 直接查看会话文件，肉眼审查内容

## 7. 开放问题 / 待技术文档决策

> **项目级技术栈**已在 [`0002`](../../decisions/0002-incubation-tech-stack/README.md) 中统一锁定。
> 以下问题属于**本需求的实现策略**，**不在本需求文档中拍板**，将在同目录下的 `design.md` 中讨论与决策：

- **Q-1 存储格式选型**：JSONL append-only / SQLite / 其他？（讨论倾向：JSONL，与 Claude Code / Gemini CLI 等主流 LLM Chat 一致；详见 `design.md`）
- **Q-2 会话元数据存放策略**：放进 JSONL 首行（不可变 meta）+ 文件 mtime（动态活跃时间）+ 不在列表展示"当前可变状态"？这是为了避免 Claude Code 已踩坑的"双数据源同步"问题
- **Q-3 SessionStore / SessionManager 分层**：是 `SessionStore`（持久化协议）+ `SessionManager`（业务编排）双层，还是单层？
- **Q-4 切换事件的形态**：是单一 `switch_event` 还是 `system_message + change_event` 一对？（讨论倾向：插一条 system 消息事件 + 一条切换标记事件，便于前端可视化）
- **Q-5 `Conversation` 与 `Session` 的关系**：`Conversation` 退化为 Session 的"运行时执行器"还是与 Session 合并？
- **Q-6 与 `NaiveContextManager` 的交互**：切 persona 后上下文里的 system 消息是只用"当前激活"还是把"切换历史"全发？
- **Q-7 `title_generator` 注入口的协议形态**：`Callable[[Session], str]` / `Protocol` / 简单 function？
- **Q-8 删除行为**：hard delete 还是 trash 目录？
