## 1. 背景与价值

### 1.1 现状

[001](../001-foundation-chat-and-memory/requirement.md) ~ [004](../004-engine-system-prompt-composer/requirement.md) 落地后，AI 朋友具备：

- 多轮文字对话与流式输出
- 跨会话恢复、persona / model 动态切换
- 按职责装配的 system prompt

但 AI 仍然**只能用训练时见过的知识作答**——一旦用户问"今天的新闻"、"最新发布的产品"、"刚出来的某条消息你听说了吗"，AI 要么承认不知道、要么硬编（比承认不知道更糟）。

### 1.2 痛点

这与 [`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 第 1 条 **"像真人，不像工具"** 直接冲突——真朋友不会反复说 "我的训练数据截止到 X 月"，而是会说 "等我帮你查查"。

[001 R-4.2.5](../001-foundation-chat-and-memory/requirement.md) 已经把"训练数据截止"列为体验红线，但只在 prompt 层口头禁止。本期从机制层面真正解决：让 AI 拥有"上网查"这种外部能力。

### 1.3 为什么"机制 + 第一个 tool"绑在一起做

纯机制（让 LLM 能调外部函数）只能用 fake tool 自测，验收没有体感；加上一个真实可用的 tool 让能力闭环、用户能直接感知"AI 朋友会上网了"。同时严格只做这一个 tool——计算器 / 文件读写 / shell 等一律不顺手做。

### 1.4 与未来需求的衔接

本期不实现的能力扩展方向（MCP 协议接入、其它具体 tool、Skill 等）将作为兄弟需求单独立项；本期的引擎层抽象需为这些方向**留扩展位**（详见 §4.1 R-4.1.6），但**不预先实现**。

## 3. 非目标（Out of Scope）

以下能力**本期明确不做**，避免范围蔓延：

- **互联网搜索之外的所有具体 tool**（计算器、文件读写、shell、本地 API 等）—— 各自单独立项
- **MCP 协议接入** —— 单独立项
- **Skill 调用能力** —— 单独立项
- **用户面向的 tool 配置 UI / 启停开关** —— 孵化期 YAGNI，未来真有用户反馈再立项
- **多用户场景的 tool 权限隔离 / 沙箱** —— 单设备单用户假设
- **AI 主动并发调用多个 tool 的并行执行优化** —— 协议层兼容即可，行为简化（详见 §7 开放问题）
- **搜索结果的本地缓存 / 去重 / 持久化索引** —— 不是搜索 tool 的本期目标

## 4. 核心需求详述

### 4.1 工具调用机制

**目标**：让 AI 朋友具备"调用工具"的能力——能在合适场景自主决定调用，把结果整合进回复，整个过程对用户体感自然、对系统可追溯。

需求点（用户/系统视角）：

- **R-4.1.1 自主判断**：AI 能根据对话内容自主决定是否调用工具——既不是用户每问一句都强制查（过度），也不是该查时偷懒不查（失职）。具体判断由 LLM 完成，不在引擎层硬编规则。
- **R-4.1.2 用户体感自然**：工具调用过程对**用户**是自然的——不暴露 API 名、provider 名、错误码、调用栈等技术细节。AI 是否在文字回复里"口头声明"自己在用工具（如"我帮你查查"），由当前 persona 决定，不在引擎层强制。
- **R-4.1.3 调用历史完整可追溯**：工具调用（含入参、结果、失败信息）必须进入会话历史。重启后能从历史中看到"那次 AI 帮我查过 X、拿到了什么、用进了哪条回复"的完整链路；与 [002](../002-engine-session-management/requirement.md) 的会话恢复语义一致。
- **R-4.1.4 失败拟人兜底**：工具调用失败（网络异常 / 限流 / 认证 / provider 错误等）时**不能让对话中断**，AI 必须用拟人话术继续（如 "我刚没查通，要不你直接告诉我？"）；与 [001 R-4.1.4](../001-foundation-chat-and-memory/requirement.md) 错误处理思路一致。
- **R-4.1.5 不发散**：工具调用循环不应陷入死循环——用户不会无限等待。具体上限值与触上限的兜底策略由 `design.md` 决定，但需求层面承诺"用户体验上有合理的等待边界"。
- **R-4.1.6 引擎层抽象稳定**：工具的"声明 / 注册 / 执行"为引擎层一等公民，对未来接入新的工具来源（如 MCP server 暴露的 tool、外部 HTTP service tool 等）应**只需新增实现、不需改动核心调用循环**。本期只交付一种来源（本地 Python 实现），但抽象不能将来源写死。

> 工具调用循环的具体形态（串行 / 并行、最大轮次、协议层细节）、工具来源的 protocol 形态、与 002 事件 schema 的具体扩展方式，由 `design.md` 决定。

### 4.2 互联网搜索能力

**目标**：交付第一项真实可用的工具能力，让 AI 朋友能查实时信息。

需求点：

- **R-4.2.1 基础检索**：能根据一个查询字符串返回若干条相关结果（至少含标题 / 摘要 / 来源 URL 三类信息），具体条数与字段格式由 `design.md` 决定。
- **R-4.2.2 有事实依据的回复**：AI **不能**把搜索结果原文直接贴给用户当回复，必须**整合后**用自己的话给出回答。这是"像真人朋友"和"像搜索引擎"的差别。
- **R-4.2.3 失败拟人兜底**：搜索失败（网络 / 配额 / provider 错误等）时按 R-4.1.4 拟人继续，**不暴露具体错误**。
- **R-4.2.4 不暴露 provider 与技术细节**：用户不应在 AI 的文字回复里看到搜索 provider 的名字（如 "通过 Tavily 搜索到..."）、API 错误码、URL 字段名等技术信息。

> 搜索 provider 的选型（Tavily / DuckDuckGo / 国内某家 / ...）、超时与重试、结果排序与裁剪策略，由 `design.md` 决定。

### 4.3 调试可观测

**目标**：让开发者能跑、能联调，能看清 AI 的工具调用决策过程。

需求点：

- **R-4.3.1 开发可见**：开发者能在调试 UI 中看到本轮"是否调用了工具 / 调用了什么 / 入参是什么 / 拿到了什么结果 / 最终回复"。与 [001 R-4.3.3](../001-foundation-chat-and-memory/requirement.md) 思路一致。
- **R-4.3.2 与用户视图分离**：调试观测**不强制对终端用户可见**。即调试 UI 看到的"tool 调用全貌"和未来形态层（桌宠 / 前端）面向用户呈现的"自然连贯回复"是两个视图，引擎层应能同时支撑两者。
- **R-4.3.3 形态不限**：调试 UI 的具体形态（带颜色的 stderr / 单独通道 / 独立日志文件 / ...）由 `design.md` 决定。

## 5. 关键体验原则

引用 [`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 三条核心原则，本期具象化为：

1. **像真人，不像工具**
   - 朋友帮你查东西的语感由 persona 决定（"等我帮你查查"或直接给整合答案都可以），但**底线是不暴露技术细节**——provider 名、API 错误码、字段名等绝不出现在 AI 文字回复里
   - **AI 应当像真人朋友一样合理判断"什么时候需要查"**——这是一条软原则，不作为 binary 验收点；该查时查、该说"我直接告诉你"时直接说，由 LLM 自主判断

2. **底层可替换，上层稳定**
   - 工具调用机制应当对"工具来源"开放（本地 / MCP / 远端服务），抽象层在本期就位，避免未来扩展时推翻核心循环
   - 搜索 provider 应当可替换（孵化期可只接一家），上层调用方不感知具体 provider

3. **失忆话术红线（在 tool 可用前提下进一步收紧）**
   - [001 R-4.2.5](../001-foundation-chat-and-memory/requirement.md) 已禁止"作为 AI / 我的训练数据截止到 X" 类话术
   - 在搜索 tool 可用的前提下，"我无法访问实时信息" 类话术**也**进入红线——既然能查就不该说

## 6. 验收标准

- **AC-1 实时信息可查**：用户在调试 UI 中提一个**明显需要实时信息**的问题（如"今天某条新闻发生了什么"、"某公司最新发布了什么"），AI 触发搜索并给出有时效性、有事实依据的回复。
- **AC-2 失败拟人兜底**：模拟搜索失败（断网 / 配置错误 key / mock provider 抛错），用户问需要查的问题，AI **不崩溃 / 不暴露错误码**，用拟人话术继续，对话不中断。
- **AC-3 跨会话可追溯**：用户在一次会话中让 AI 查了某事并获得回答；正常退出 + 重新启动后通过 `--resume` 打开同一会话，能从会话历史中看到完整的工具调用链路（调了什么、拿到了什么、AI 怎么回复的）。
- **AC-4 失忆话术红线**：在搜索 tool 可用的前提下，AI **不应**出现 "我的训练数据截止到..."、"作为 AI 我无法访问实时信息"、"我没有联网能力" 等话术。
- **AC-5 调试可观测**：开发者在调试 UI 中能看到"本轮是否调了 tool / 调了什么 / 拿到什么结果"等观测信息，且这些信息**默认不出现**在面向终端用户的回复文字里。
- **AC-6 既有行为不退化**：001 ~ 004 已交付的所有行为（多轮对话、流式输出、跨会话恢复、persona / model 切换、system prompt 三段装配等）在本期落地后**全部继续可用**，无回归。

## 7. 开放问题 / 待技术文档决策

> [`0002`](../../decisions/0002-incubation-tech-stack/README.md) 中已锁定的项目级技术栈不重复。
> 以下问题属于**本需求的实现策略**，**不在本需求文档中拍板**，将在同目录下的 `design.md` 中讨论与决策：

- **Q-1 搜索 provider 选型**：Tavily / DuckDuckGo / SerpAPI / 国内 provider（Bocha / 智谱搜索 / ...）/ 自部署 SearXNG？需结合 [`0002 §3.13`](../../decisions/0002-incubation-tech-stack/README.md) 的"零障碍访问、零支付门槛"风格选择
- **Q-2 引擎层 Tool / ToolRegistry 抽象**：`Tool` Protocol / `ToolRegistry` 的具体接口形态？工具来源（本地 / MCP / 远端）如何在抽象上留位？
- **Q-3 工具调用循环的协议层与上限**：LLM 一次返回多个 tool_call 时串行执行还是并行？最大循环轮次（防发散）取多少？触达上限时的兜底动作？
- **Q-4 与 002 事件 schema 的扩展**：tool_call / tool_result 作为新事件类型加入 [002](../002-engine-session-management/requirement.md) 的 append-only JSONL 事件流，schema 字段如何定？对现有事件流的向后兼容性如何保证？
- **Q-5 与 004 system prompt 的关系**：是否需要新增一个 prompt section 提示 LLM "你有哪些工具可用 / 何时该调"？还是依赖现代 LLM 看到 `tools` 参数就自己判断？
- **Q-6 引擎对外接口的多通道扩展**：当前 `Conversation.stream()` 只 yield 文本字符串。tool 调用过程要让"开发可观测 + 用户体感连贯"同时成立，需要怎么扩接口（多类型事件流？旁路日志？双方法？），且对现有 CLI 影响最小？
- **Q-7 CLI 调试观测的具体形态**：带颜色 stderr / 独立日志文件 / inline 标记 / ...？需要多详细？
- **Q-8 失败兜底话术的承载位**：拟人兜底话术属于 [001 fallbacks 池](../001-foundation-chat-and-memory/design.md)的扩展，还是 tool 模块自带一组？
