# 008 · 引擎层长期记忆

> Engine-Level Long-Term Memory
>
> 把 [001](../001-foundation-chat-and-memory/requirement.md) 阶段 2 预留、却一直没真正交付的记忆能力落地：让 AI "真的记得" 用户。

---

## 1. 背景

### 1.1 当前现状

记忆是 [`0001 产品愿景`](../../decisions/0001-product-vision-and-roadmap/README.md) 里 **M2 · 基础记忆** 的核心地基，但至今只搭了"插座"、没接"电器"：

- [`001 design §6`](../001-foundation-chat-and-memory/design.md) 预留了完整的记忆接入点：`Conversation.__init__(memory=None)`、`ContextManager.build_messages(extra_context=)`、`Memory(Protocol).retrieve`、`Message.meta`。
- `memory/` 模块至今仅 `Memory(Protocol)` 占位，`retrieve` 未实现，`Conversation` 实际注入 `None`。
- [`002 引擎层会话管理`](../002-engine-session-management/requirement.md) 已把会话上移为引擎一等公民，`JsonlSessionStore` 落地。会话流是 **append-only 事件流**，002 §1.3 已明确它是"未来 memory 抽取的原始数据源"。

→ 结论：**地基里"记忆抽取的数据源"已经就位，但"记忆"本身从未被实现**。001 列出的记忆验收标准（跨会话记忆 / 不失忆 / 可观测）至今没有真正交付，这正是 008 要补的缺口。

### 1.2 为什么现在做

引用 [`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 第 2 条 **"记忆是第一护城河"**。

对话（001）、会话（002）、人格（003）、system prompt（004）、工具（005）、bridge（006）、语音（007）等地基都已就位，记忆是最后一块尚未填上的核心地基。原始对话数据源（会话流）已经稳定，可以开始做"从对话中抽取 + 跨会话召回"。

### 1.3 与已有需求的边界

| 概念 | 定位 | 与 008 的关系 |
|---|---|---|
| Conversation / 对话（001） | 跑一轮对话的运行时执行器 | 008 通过 001 §6 预留的 `memory` 参数与 `extra_context` 接入，**不改其公开接口** |
| Session / 会话（002） | 一段连续对话的原始载体（append-only 事件流） | 008 把它当作**只读数据源**：记忆从会话流里抽取，不回写会话 |
| Memory / 长期记忆（008，本期） | 从对话中抽出的、可跨会话检索的语义化记忆 | 本期要做的 |

**Session ≠ Memory**（沿用 002 §1.3）：Session 是"原始对话"，Memory 是从对话中**抽出**的认知。两者平行、互不依赖，存储介质可不同。

> 本需求的思考过程沉淀在 [`docs/explorations/memory`](../../explorations/memory/README.md)；该探索是过程笔记，**结论的 single source of truth 以本 `requirement.md` 与后续 `design.md` 为准**。

## 2. 本期范围（In Scope）

本期按 exploration §1 归纳的**四类记忆使用场景**组织交付物：

| 场景 | 一句话 | 本期 |
|---|---|---|
| 维护写入 | 从对话中抽取值得记的信息并持久化 | **做** |
| 显性引用 | 用户嘴里说出来要 agent 想起，能召回 | **做** |
| 隐性应用 | 用户没明说，agent 自动带上相关上下文 | **做** |
| 连续感 | 让 agent 像"一直是同一个朋友" | **做** |

具体落到：

- **抽取与持久化**：从对话流中识别"值得记住的信息"，落盘、进程重启不丢。
- **跨会话召回**：新会话里能召回此前会话沉淀的记忆。
- **默认自动注入**：每轮对话默认自动召回相关记忆作为上下文，不依赖模型主动决定"去查记忆"。
- **调试可观测**：开发者能看到本轮召回了哪些 / 写入了哪些记忆。
- **按预留点接入**：仅通过 [`001 design §6`](../001-foundation-chat-and-memory/design.md) 已承诺的预留点接入，不破坏 001–007 已交付的对外接口。

## 3. 非目标（Out of Scope）

以下能力本期**明确不做**；其中多数已在 [exploration](../../explorations/memory/README.md) 中确认"schema 预留位、v1 不实现行为"：

- **Reflection（反思）**：扫多条记忆找模式、产出更抽象的认知。v1 只在 schema 留 `source` 字段位（exploration §3.4），不实现异步反思管线。
- **inspect / forget**：用户翻看 agent 的记忆 / 让 agent 主动忘掉某事。未来必做（信任、隐私、纠错），v1 不暴露方法，schema 字段预留（exploration §5）。
- **多 user / 多 persona 行为**：schema 预留 `owner_user_id` + `persona_id`，v1 锁死单一 `(user, persona)`，不暴露多 user / persona 行为（exploration §5 / §6）。
- **M9 强记忆系统升级**：更强的记忆架构属未来独立需求，见 [`0001`](../../decisions/0001-product-vision-and-roadmap/README.md)。
- **改动会话存储 / 回写会话**：008 只读会话流，不改 002 的存储格式、不向会话流写记忆。
- **冷启动话术**：第一次见面怎么自我介绍、怎么问基础问题，归 persona / 编排，不在记忆职责内（exploration §5）。

## 4. 核心需求详述

以下需求点（R-级）按四类场景组织，只讲"做什么 / 做到什么效果"；具体抽取策略、存储 schema、召回算法、注入位置等"怎么做"由 `design.md` 决定。

### 4.1 记忆写入 / 抽取（维护写入路径）

**目标**：让"值得记住的信息"能从对话里被识别出来并长期留存。

- **R-4.1.1 识别值得记的信息**：能从对话中识别事实、偏好、关系、重要事件、与 AI 之间发生过的事（如"用户养了一只叫 Tom 的猫"、"用户讨厌香菜"）。
- **R-4.1.2 持久化不丢**：记忆必须落盘，进程重启后不丢。
- **R-4.1.3 以会话流为数据源**：写入以 002 的会话流作为事实来源与溯源指针，**不重复存储原始对话原文**。
- **R-4.1.4 来源权重有别**：用户说过的（user-said）比 AI 说过的（agent-said）写入权重高一档（exploration §5）。
- **R-4.1.5 状态更新与纠正**：当用户改口（如"我换工作了"）或纠正错记时，新信息应能更新旧记忆，而**不是让相互矛盾的事实长期并存、导致召回自相矛盾**（达到何种程度、冲突如何解决由 `design.md` 定，见 Q-7）。

### 4.2 跨会话召回（显性引用）

**目标**：用户嘴里点名要 agent 想起的事，能被准确召回。

- **R-4.2.1 显性提起可召回**：用户在新会话里显式提起旧事（"上次你说的那个…"、"我之前提过 X 没？"、"你还记得 Y 吗？"），能召回对应记忆。
- **R-4.2.2 高精准**：用户给了明确锚点时，记忆中存在即应被召回。允许"记得但细节模糊"，**不允许在确有记录时表现为"没印象"**（exploration §1.1）。

### 4.3 默认自动注入相关上下文（隐性应用）

**目标**：用户没明说，但 agent 能自动带上相关上下文——这是 exploration 认定的**最高频**记忆使用方式。

- **R-4.3.1 默认每轮自动召回**：召回是对话编排的**默认行为**，每轮自动跑一次，**不依赖模型自己决定"我要查记忆"**（模型未必能识别"Tom 是需要查的实体"，依赖模型决定会让隐性应用直接失效）。
- **R-4.3.2 模型无感**：被注入的记忆对模型表现为普通上下文；模型**不应在回复里暴露"让我查一下记忆"**之类的措辞（一秒变 AI 客服）。
- **R-4.3.3 宽召回但抑制噪声**：隐性应用下"捞错"代价高于"漏召回"（塞进不相关的事像"突然多嘴的 AI"）；召回策略应在"宽召回"与"抑制噪声"之间取平衡（exploration §1.2）。

### 4.4 连续感

**目标**：让 agent 感觉"一直是同一个朋友"，而不是每次重启都像刚认识。

- **R-4.4.1 跨会话连续**：新会话开场不应表现得像"我们刚认识"。
- **R-4.4.2 时间感可用**：记忆应能提供"上次聊到 X""多久没聊了"这类带时间维度的信息，供上层自然带出（**具体话术归 persona**，记忆只负责提供可用信息）。
- **R-4.4.3 关键事实常驻**：被标记为"必须每次都进上下文"的关键事实（如用户姓名、核心关系），每轮都应可用（exploration §3.2 的 `pinned` 语义）。

### 4.5 调试可观测

**目标**：开发者能看清记忆系统每轮做了什么，便于联调（承接 001 R-4.3.3 / AC-5）。

- **R-4.5.1 召回可见**：能看到本轮召回了哪些记忆（含来源指针、得分等便于排查的信息）。
- **R-4.5.2 写入可见**：能看到本轮新写入 / 更新了哪些记忆。
- **R-4.5.3 如实反馈**：召回为空时记忆系统**如实返回"未找到"，不编不藏**；话术兜底由 persona 负责（exploration §5）。

## 5. 关键体验原则

引用 [`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 三条原则，本期重点服从：

1. **记忆是第一护城河**：记忆的健壮性 > 任何花哨能力。宁可形态简陋，不可"确有记录却失忆"。
2. **像真人，不像工具**：记忆注入对模型无感，不暴露"我在查记忆 / 我有记忆模块"；"失忆话术"不由记忆系统硬禁，而由 persona 的 fallback 用真人化措辞兜底（呼应 [`001 design §4.4.4`](../001-foundation-chat-and-memory/design.md) 否决"失忆话术禁令"）。
3. **底层可替换，上层稳定**：记忆的存储介质与检索实现（关键词 / FTS / 向量等）应可替换；对外仅通过 [`001 design §6`](../001-foundation-chat-and-memory/design.md) 的稳定预留点接入，不破坏已承诺接口。

## 6. 验收标准

> 验收标准的写法**不绑定具体检索实现**：无论 design 选关键词 / FTS 还是向量检索，以下 AC 都应可被满足（v1 不把"向量检索"列为验收门槛，见 Q-2）。

- **AC-1 跨会话显性召回**（细化自 [`001 AC-3`](../001-foundation-chat-and-memory/requirement.md)）：
  - 用户在一次会话中告诉 AI 关键事实（如"我叫小明，养了一只猫叫 Tom"）。
  - 关闭进程、重新启动。
  - 在新会话中以自然方式显式提起（"我那只猫还记得吗"），AI 能正确召回且表达自然。
- **AC-2 隐性应用**（人工观察型）：给定典型样例（如用户说"今天又见到 Tom 了" / 问"晚上吃啥推荐"），在用户**没有明说要查记忆**的情况下，AI 能自动带出相关旧信息（知道 Tom 是猫 / 想起用户讨厌香菜）。召回为概率性，**验收以人工确认"表现合理、无明显'本该想起却没想起'"为准，不要求 100% 命中**。
- **AC-3 连续感**：相隔两段独立会话后，第二段开场 AI 表现出"认识用户"的连续感（不重新自我介绍、能自然带出此前主题或"好久没聊了"），不像刚认识。
- **AC-4 不失忆 + 职责边界**（细化自 [`001 AC-4`](../001-foundation-chat-and-memory/requirement.md)）：
  - 记忆中**确实存在**相关信息时，记忆系统必须召回到它（这是 008 的硬保证）。
  - 确实没有记录时，记忆系统如实返回"未找到"，话术由 persona fallback 负责。
  - 任何情况下**不得出现暴露 AI / 模型身份**的话术（"作为 AI"、"我没有记忆能力"等）；但"嗯…我有点记不太清了，你再说说？"这类真人化措辞**不算违反**。
- **AC-5 调试可观测**（= [`001 AC-5`](../001-foundation-chat-and-memory/requirement.md)）：开发者能在调试入口看到本轮召回了哪些记忆 / 写入了哪些记忆。
- **AC-6 持久化跨进程不丢**：写入的记忆在 `kill` 进程后重启仍可被召回。
- **AC-7 不破坏既有接口**：001–007 已交付的对外行为 / 接口不退化；记忆仅通过 [`001 design §6`](../001-foundation-chat-and-memory/design.md) 的预留点接入。

## 7. 开放问题 / 待技术文档决策

> **项目级技术栈**（语言、LLM Provider、存储中间件、目录结构等）已在 [`0002`](../../decisions/0002-incubation-tech-stack/README.md) 锁定，本节不重复。
> 以下属**本需求的实现策略**，**不在本文档拍板**，将在同目录 `design.md` 中讨论（多数对应 [exploration §6 Still Open](../../explorations/memory/README.md)）：

- **Q-1 记忆分层 schema**：episodic / semantic / pinned 的具体字段、类型、索引（exploration §3.1 / §3.2 / §5 预留位为基础）。
- **Q-2 召回策略**：关键词 / FTS vs 向量检索（若用向量，0002 §3.16 预备 Chroma + BGE-small-zh）；v1 是否启用向量——**requirement 不锁，design 定**。
- **Q-3 召回打分公式**：relevance × importance × time-decay 的具体形式（参考 Generative Agents 但要简化）。
- **Q-4 重要性打分由谁打**：LLM / 规则 / 混合。
- **Q-5 抽取管线**：抽取 prompt 设计 + 抽取时机（exploration §3.4 倾向"每段对话结束后同步抽取"）。
- **Q-6 记忆注入位置**：走 001 的 `extra_context` 路径，还是并入 [`004 system prompt composer`](../004-engine-system-prompt-composer/requirement.md)。
- **Q-7 状态更新 / 冲突解决**：用户改口（"我换工作了"）/ 纠错时如何更新旧记忆、如何避免矛盾事实并存（对应 R-4.1.5）。
- **Q-8 多 persona 粒度**：semantic（user 维度事实）跨 persona 共享 vs episodic 按 persona 隔离（v1 锁单一，但 schema 要预留得对，exploration §6 第 1 条）。
- **Q-9 Reflection 触发信号**：v1 不实现，但需回答"什么信号触发开始反思"以稳住 `source` 字段（exploration §6 第 5 条）。
