# 009 · 引擎层上下文管理

> Engine-Level Context Management
>
> 补上 [`001`](../001-foundation-chat-and-memory/requirement.md) 以来一直留白的"问题 B：跨时间演化"——让会话在 context window 预算内长期跑下去而不丢关键信息，不再撑爆即抛错。

---

## 1. 背景

### 1.1 当前现状

[`context-management exploration §1`](../../explorations/context-management/README.md) 把"上下文管理"拆成两个问题：

- **问题 A · 单次组装**：`system + memory + history + new_input + tools` 怎么拼。已由 [`001 design`](../001-foundation-chat-and-memory/design.md) 的 `build_messages` 完成，是"静态快照"，本期**不动**。
- **问题 B · 跨时间演化**：随轮数增长，`history` 单调膨胀，迟早撑爆 context window。这是当前真正的缺口。

现状的 `NaiveContextManager` 全发不截断——把问题 B 当作"不存在"：撑爆时直接抛 `LLMBadRequestError`，由 CLI 给友好提示。`ContextManager` 已是 Protocol，注释预告了 `Fifo` / `TokenBudget` / `Summarizing` 三种策略，但**都没实现**。

> 注意与 [`008 长期记忆`](../008-engine-memory/requirement.md) 区分：008 做的是**跨会话长期记忆**（事实/情节蒸馏，独立库带外召回）；本需求只管**单会话内 history 怎么度量、裁剪、压缩**。两者平行、互补，见 §5。

### 1.2 为什么现在做

对话（001）、会话（002）、人格（003）、system prompt（004）、工具（005）、bridge（006）、语音（007）、记忆（008）等地基均已就位。其中工具调用（005）让单轮上下文体积显著上升，长会话撑爆窗口从"理论风险"变成"很快会撞到的现实"。`NaiveContextManager` 的"撑爆即抛错"不再可接受，必须补上跨时间演化能力。

### 1.3 与已有需求的边界

| 概念 | 定位 | 与 009 的关系 |
|---|---|---|
| `build_messages` / 单次组装（001） | 把各成分拼成一次 LLM 调用的"静态快照" | 009 不改其职责，只替换其中 `history` 的来源策略 |
| `ContextManager`(Protocol)（001） | 决定"这一轮 history 发什么"的可插拔点 | 本期落地它的真实实现（度量 + 兜底 + 摘要） |
| Session / 会话流（002） | append-only 事件流，原始对话载体 | 009 **只读不写**：压缩只作用于发给 LLM 的上下文，不改会话流落盘内容 |
| Memory / 长期记忆（008） | 跨会话语义记忆，带外召回 | 平行互补，构成被压缩历史的第二张安全网（§5），但**不做强协同** |

> 本需求的思考过程沉淀在 [`docs/explorations/context-management`](../../explorations/context-management/README.md)；该探索是过程笔记，**结论的 single source of truth 以本 `requirement.md` 与后续 `design.md` 为准**。

## 2. 本期范围（In Scope）

本期按探索 §2 的分层蓝图交付 L1 / L2，并按**三个里程碑分阶段落地与验收**（探索 §6 的落地节奏）：

| 里程碑 | 一句话 | 对应蓝图层 |
|---|---|---|
| **M1 度量与预算** | 能回答"当前占多少 token、还剩多少窗口、超没超阈值" | L1 |
| **M2 防爆窗兜底** | 摘要不可用时，零 LLM 成本的 FIFO 截断保证不爆窗 | L2 · 路 B |
| **M3 摘要压缩** | 主策略：结构化摘要把长历史蒸馏成 summary，腾出窗口 | L2 · 路 A |

三者关系：**正常走 M3 摘要压缩；摘要链路跳闸（circuit breaker）后退化到 M2 的 FIFO 截断防爆窗**。M1 是 M2/M3 的前置——不度量就无法判断"超没超预算"。

具体落到：

- **M1 token 度量**：让系统能度量"当前上下文 token 占用 / 剩余窗口"。需 `LLMClient` 透出真实 `usage`（当前流式/非流式都未透出，见 §7 Q-1），并用 hybrid 估算（真实 usage 为锚点 + 新增消息字符粗估并保守 padding）。
- **M1 触发阈值**：阈值**按所用 model 的 context window 动态推导**（不写死固定值），缓冲量按窗口比例计算（探索 §4 的"窗口 − 输出预留 − 缓冲"相对公式）。
- **M2 FIFO 兜底**：作为摘要不可用时的安全兜底，零额外 LLM 成本，保证长会话不再抛 `LLMBadRequestError`。
- **M3 摘要压缩**：超阈值时把旧历史蒸馏为结构化 summary 注入上下文，腾出窗口（探索 §4 工程要点：user 消息逐字保留、`<analysis>` 草稿纸、摘要调用禁用 tool、circuit breaker 连续失败放弃）。
- **M3 摘要持久化**：摘要结果作为新增的 `compaction` 折叠点事件**落盘**，进程重启 / `--resume` 后可复用、不必重算；原始事件保留不动（与 §3 非目标边界一致）。
- **可观测**：开发者能看到本轮是否触发了压缩 / 走了兜底 / 当前 token 估算与阈值。

## 3. 非目标（Out of Scope）

以下本期**明确不做**（多数已在探索中确认）：

- **L3 轮内 tool 结果爆炸控制**（探索 §2/§4）：单条 tool 结果（web search 一大坨、读文件几千行）瞬间顶满窗口的清理/摘要。纳入蓝图、**本期不实现**，留后续独立需求。
  - 注意边界：本期会让**工具循环续轮**也经过上下文管理（见 R-0.3），但那是"对整轮历史做预算检查/裁剪/压缩"（L2 完整性）；**砍单条巨型 `tool_result` 的体积**才是 L3，本期不做。
- **选择性保留（按重要性留/丢）**（探索 §3）：比 FIFO 聪明但需打分逻辑，本期不做。
- **与长期记忆的强协同**（探索 §5）：不做"压缩前先确认旧历史已沉淀进 memory"的耦合机制。现有每轮 `observe` + 两张安全网已是尽力而为的交接（§5）。
- **删除 / 修改原始会话事件**：摘要压缩**不删除、不修改**任何已落盘的原始 user / assistant / tool 事件——它们仍是 memory 抽取的完整数据源（002 §1.3）。
  - 注意：本期**会新增** `compaction` 折叠点事件来持久化 summary（见 §2 M3 持久化 / R-3.7）。这是在 append-only 事件流上**叠加**一个折叠点，不动原始事件、不改既有事件 schema；`Session.messages`（memory 读的原始全量视图）行为不变。
- **改动 `build_messages` 的单次组装职责**（问题 A）：本期只替换 `history` 的来源策略，不重构组装逻辑。

## 4. 核心需求详述

以下需求点（R-级）按里程碑组织，只讲"做什么 / 做到什么效果"；具体估算系数、保留窗口大小、摘要 prompt 模板、circuit breaker 阈值等"怎么做"由 `design.md` 决定。

### 4.1 M1 · 度量与预算

**目标**：让系统在每轮发起 LLM 调用前，能判断"当前上下文是否逼近 / 超出预算"。

- **R-1.1 token 占用可度量**：能给出"当前要发出的上下文"的 token 估算值。允许是估算而非精确，但应**偏保守（偏上界）**，宁可早触发不可漏触发。
- **R-1.2 真实 usage 透出**：`LLMClient` 在主对话路径（流式）与非流式路径都能透出 API 返回的真实 token `usage`，作为估算锚点。
- **R-1.3 阈值随 model 动态推导**：触发阈值必须按当前所用 model 的 context window 推导，缓冲量按比例计算，**不得写死固定 token 值**（不同 provider/model 窗口差异极大：8K / 32K / 128K / 200K / 1M）。
- **R-1.4 度量可观测**：开发者能看到本轮的 token 估算值、当前 model 的有效窗口与触发阈值。

### 4.2 M2 · 防爆窗兜底（路 B / FIFO）

**目标**：在摘要链路不可用时，仍保证长会话不会因撑爆窗口而抛错中断。

- **R-2.1 不再撑爆即抛错**：超出窗口预算时，系统能自动裁剪 history 使请求落回预算内，而**不是直接抛 `LLMBadRequestError` 中断对话**。
- **R-2.2 兜底定位明确**：FIFO 截断仅作为**摘要不可用时的安全网**（M3 跳闸 / 未启用），零额外 LLM 成本；正常情况下由 M3 主导。
- **R-2.3 保留最近上下文**：截断时优先丢最老的消息、保留最近的对话，使当前轮仍有可用的近期上下文（保留窗口大小由 `design.md` 定，见 Q-3）。

### 4.3 M3 · 摘要压缩（路 A / 主策略）

**目标**：在超阈值时把长历史蒸馏为结构化 summary，在腾出窗口的同时尽量不丢关键信息。

- **R-3.1 超阈值触发压缩**：达到 M1 推导出的阈值时，自动把旧历史压缩为一条结构化 summary 注入后续上下文。
- **R-3.2 用户意图不漂移**：摘要中**用户原话关键信息应被如实保留**，避免转述导致用户意图/事实漂移（探索 §4：cc 的"user 消息逐字保留"）。
- **R-3.3 摘要质量保障**：允许借鉴 `<analysis>` 草稿纸等手段先梳理再输出，最终只把 summary 留进上下文（探索 §4）。
- **R-3.4 摘要调用受控**：摘要 LLM 调用应禁用工具、单轮文本输出，避免摘要过程触发副作用（探索 §4）。
- **R-3.5 失败不进入死循环**：摘要连续失败时有 circuit breaker 放弃机制，跳闸后退化到 M2 兜底，不反复重试拖垮对话（探索 §4）。
- **R-3.6 压缩可观测**：开发者能看到本轮是否触发压缩、压缩前后 token 变化、是否跳闸退化到兜底。
- **R-3.7 摘要持久化且不动原始**：摘要结果须持久化（重启 / `--resume` 后可复用、不重复付费），且**不删除 / 不修改**任何原始会话事件——通过在 append-only 事件流上新增 `compaction` 折叠点实现；memory 抽取读到的原始全量视图不受影响。

> ⚠️ M3 会真实调用外部 LLM（摘要）。其触发与验证均受 `llm-api-confirm` 规则约束。

### 4.4 通用

- **R-0.1 可插拔不破坏接口**：通过 `ContextManager`(Protocol) 的既有预留点接入，不破坏 001–008 已交付的对外行为与接口。
- **R-0.2 策略可配置**：能在"不裁剪 / FIFO 兜底 / 摘要压缩"之间选择，便于分里程碑上线与回退。
- **R-0.3 覆盖所有 LLM 上下文入口**：上下文管理（度量 / 兜底截断 / 摘要压缩）必须覆盖**每一次发往 LLM 的上下文组装**，不只首轮——含工具调用循环的**续轮（turn>0）**与**触上限兜底收尾**。现状 `Conversation` 仅首轮经过 `ContextManager`，续轮（`_build_openai_messages_continuation`）与兜底（`_finalize_on_tool_loop_limit`）直接全量拼 `session.messages`，使防爆窗在带工具的长会话下出现漏洞，本期需收口。统一入口的接口形状（`build_messages` 现要求 `new_user_input`、续轮无新输入）如何调整由 `design.md` 定（见 Q-9）。

## 5. 与长期记忆（008）的关系

被压缩/截断的历史，关键信息有**两张互补安全网**（探索 §5）：

1. **安全网 1 · 压缩 summary**：留在对话上下文里（in-band）。
2. **安全网 2 · memory 已抽取的事实**：008 的独立库，按相关性带外召回。

链路现状：每轮成功结束 → `Conversation._observe_turn` → `memory.observe` 非阻塞入队 → 串行抽取，即**每轮都自动交接**。因此本期 **不做"压缩前先确认已沉淀"的强协同**——接受现有 observe 的尽力而为即可。需明确这份交接是**尽力而为、非无损**（异步时差 / 有损抽取 / 崩溃丢队列，008 design 已声明 v1 接受）。

## 6. 验收标准

> 验收按里程碑分组。摘要为概率性能力，相关 AC **不要求 100% 命中**，以"人工观察型"判定（参照 [`008 AC-2`](../008-engine-memory/requirement.md)），**由开发者（agent）通过实际对话观察验证**。涉及真实 LLM 调用的验证遵守 `llm-api-confirm`。

### M1 · 度量与预算

- **AC-1.1 usage 透出**：一次真实 LLM 调用后，系统能拿到 API 返回的 token `usage`（流式与非流式路径均可）。
- **AC-1.2 阈值随 model 变**：切换到不同 context window 的 model 时，触发阈值随之变化、由窗口动态推导，无写死固定值。
- **AC-1.3 度量可观测**：调试入口能看到本轮 token 估算、有效窗口、触发阈值。

### M2 · 防爆窗兜底

- **AC-2.1 长会话不再抛错**：在仅启用 FIFO 兜底的配置下，持续多轮对话直到超出窗口预算，系统自动裁剪使对话继续，**不抛 `LLMBadRequestError` 中断**。
- **AC-2.2 保留近期上下文**：触发兜底截断后，当前轮仍能基于最近若干轮对话正常应答。
- **AC-2.3 工具循环续轮不爆窗**（对应 R-0.3）：在带工具的长会话中（多轮工具调用累积大量上下文），**续轮与触上限兜底收尾同样经过上下文管理**，超预算时自动裁剪/压缩，不因续轮绕过管理而抛 `LLMBadRequestError`。

### M3 · 摘要压缩

- **AC-3.1 触发压缩腾窗**：长会话达到阈值时自动触发摘要压缩，压缩后上下文 token 占用显著下降、对话可继续。
- **AC-3.2 关键信息不漂移**（人工观察型）：压缩发生后，此前用户明确给过的关键事实/意图仍能被后续轮次正确使用，无明显"压缩后就忘了用户说过什么"。
- **AC-3.3 跳闸退化**：摘要连续失败触发 circuit breaker 后，自动退化到 M2 兜底，对话不中断、不陷入重试死循环。
- **AC-3.4 压缩可观测**：调试入口能看到是否触发压缩、压缩前后 token 变化、是否跳闸退化。
- **AC-3.5 摘要持久化复用**（对应 R-3.7）：一次会话触发摘要压缩后，`kill` 进程并重新 `--resume` 打开，能复用已落盘的 summary（不重新摘要）；且原始 user / assistant / tool 事件在文件中完好无缺。

### 通用

- **AC-0.1 不破坏既有接口**：001–008 已交付的对外行为/接口不退化；上下文管理仅通过 `ContextManager`(Protocol) 预留点接入。

## 7. 开放问题 / 待技术文档决策

> **项目级技术栈**（语言、LLM Provider、存储、目录结构等）已在 [`0002`](../../decisions/0002-incubation-tech-stack/README.md) 锁定，本节不重复。
> 以下属**本需求的实现策略**，**不在本文档拍板**，将在同目录 `design.md` 中讨论（多数对应探索 §7 Open Questions）：

- **Q-1 `LLMClient` usage 透出方式**：非流式 `complete`（litellm `response.usage` 现成）与流式 `stream`（需 `stream_options={"include_usage": True}` + `LLMTurnDone` 加 usage 字段）的具体改法与改动面。
- **Q-2 缓冲量算法**：固定 token vs 窗口比例（倾向比例化，但需验证小窗口模型 8K 下比例是否合理，探索 §7）。
- **Q-3 FIFO 保留窗口**：按条数还是按 token 预算保留最近上下文（探索 §7）。
- **Q-4 token 增量字符系数**：cc 的 `/4` + `4/3` padding 对中文/多语言是否需调整（探索 §7）。
- **Q-5 摘要 prompt 模板**：是否采用 cc 的 9 段式结构、`<analysis>` 草稿纸的具体形态。
- **Q-6 circuit breaker 阈值**：连续失败几次跳闸、跳闸后是否/何时恢复重试。
- **Q-7 策略装配与配置**：`Naive` / `Fifo` / `Summarizing` 三种实现如何选择与注入，默认用哪个。
- **Q-8 摘要触发与 M2 截断的预算账本**：M3 触发点与 M2 截断点如何共享同一套"预算"概念（探索 §7 提及未来与 L3 统一的账本雏形）。
- **Q-9 统一 LLM 上下文入口的接口形状**（对应 R-0.3）：`build_messages` 现要求 `new_user_input`，但工具循环续轮 / 兜底收尾没有新用户输入。是把 `new_user_input` 改可选、还是抽一个不带新输入的预算化组装入口，由 `design.md` 定。此外兜底收尾（`_finalize_on_tool_loop_limit`）需要在末尾追加一条临时 system 指令，这一"trailing system"维度也应纳入统一入口设计，避免兜底再手拼消息（消除与续轮组装的重复）。
- **Q-10 compaction 事件 schema 与折叠投影**（对应 R-3.7）：`compaction` 事件的 payload 结构（summary 文本、覆盖到哪条事件的指针）、`Session` 折叠视图（如 `messages_for_context()`）的派生规则、多次压缩如何叠加、向后兼容（老文件无该事件时自然退化为全量），由 `design.md` 定。
