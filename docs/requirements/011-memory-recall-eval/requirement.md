# 011 · 记忆召回质量评测机制

> Memory Recall Evaluation Harness
>
> 为 [008 引擎层长期记忆](../008-engine-memory/requirement.md) 提供一套可复用、可追溯的**召回质量评测机制**：把外部成熟 benchmark 的对话灌进 memory 的真实写路径，再用 benchmark 问题观察召回。

---

## 1. 背景

### 1.1 当前现状

[008](../008-engine-memory/requirement.md) 已把记忆系统（抽取 + 召回）落地，但它的召回质量**至今只能靠人工冒烟验证**——008 `progress.md` 记录的验收方式就是"手动问一句，看 AI 记不记得 Tom 是橘猫"。这种方式有两个问题：

- **不可复用**：每次都要人肉构造对话、肉眼判断，无法重复、无法横向对比。
- **不可追溯**：memory 的抽取 / 召回策略会持续演进（向量召回、reflection 等都在 008 的预留位上），但没有一个稳定的观察手段能看出"这次改动让召回变好还是变差"。

与此同时，业界已有成熟的 agent-memory 评测 benchmark（LoCoMo、LongMemEval、PerLTQA 等），提供了现成的长对话 + 配套问答，可以作为评测素材。

→ 结论：**记忆系统已落地，但缺一套把它接到外部 benchmark 上、可复用观察召回的机制**。这正是 011 要补的缺口。

### 1.2 为什么现在做

[`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 把"记忆是第一护城河"列为不可妥协的核心。护城河要守得住，前提是**能被持续度量**——否则记忆策略一改，好坏全凭手感。008 主体已稳定，正是补上"可复用评测机制"的时机。

### 1.3 与已有需求的边界

| 概念 | 定位 | 与 011 的关系 |
|---|---|---|
| Memory / 长期记忆（008） | 从对话抽取、可跨会话召回的记忆系统 | 011 是它的**外部消费方/观察者**：只通过其公共接口（`observe` / `retrieve`）灌入与召回，**不改 memory 任何代码** |
| 会话 / Session（002） | 真实运行时的对话事件流 | 011 **不碰**：评测对话来自外部 benchmark，不经过会话流 |

**本需求交付的是"评测机制"本身，不是 memory 的召回表现。** memory 测得好不好是 008 的事；011 只保证"有一套能跑通的机制去测它"。

## 2. 本期范围（In Scope）

本期交付一套**可跑通的召回评测机制**，包含：

- **多 benchmark 可插拔加载**：把外部 benchmark 的原始数据归一成统一的评测样本结构；首期接入原生中文的 **PerLTQA**（对话子集）+ 英文 **LoCoMo**（基线对照）。
- **灌入 memory 真实写路径**：把 benchmark 的对话按段投影、经 `observe` 走真实抽取落库（而非绕过抽取直接塞数据），保证测的是 memory 的完整链路。
- **召回与展示**：用 benchmark 问题走 `retrieve`，把召回到的记忆条目结构化展示，供人评估。
- **判分扩展点**：为未来的自动判分（LLM-as-judge）预留稳定接口，本期只放空实现（不打分）。
- **机制自测**：解析 / 适配等核心逻辑有单测覆盖；脚本一键可跑；真实 LLM 调用与单测隔离。

> 本期验收的对象是**机制能不能跑通**，不是 memory 召回到什么准确率。

---

## 3. 非目标（Out of Scope）

以下本期**明确不做**：

- **召回质量的量化目标**：不设定 / 不验收任何"召回准确率达到 X%"的指标。本期不回答"memory 测得好不好"，只回答"机制能不能测"。
- **LLM-as-judge 自动判分**：仅留扩展点，不实现量化打分（属未来演进，见 §7）。
- **PerLTQA 非对话子集**：profile / 社会关系 / events 等结构化记忆，与 memory"从对话抽取"的范式不符，本期不取。
- **纯检索口径评测**：绕过抽取、直接把记忆塞进存储只测检索，是另一种评测目标，本期不做。
- **向量召回的对比评测**：008 的向量召回尚未实现，本期不为其设专门口径。
- **自动化回归门禁 / CI 接入**：不把评测纳入 `scripts/check` 等强制门禁（评测要真实 LLM、成本与稳定性都不适合做门禁）。
- **改动 memory 代码**：011 只消费 memory 公共接口，不改其实现。

## 4. 核心需求详述

以下需求点（R-级）只讲"做什么 / 做到什么效果"；具体的包结构、数据归一化、适配映射、判分接口形态等"怎么做"由 `design.md` 决定。

### 4.1 benchmark 加载

- **R-4.1.1 统一样本结构**：不同 benchmark 的原始格式被归一成一套统一的评测样本表示，下游适配 / 编排 / 展示不感知具体 benchmark。
- **R-4.1.2 可插拔扩展**：新增一个 benchmark = 新增一个加载器，不改动适配 / 编排 / 展示主干。
- **R-4.1.3 防御式解析**：脏数据 / 缺字段的条目跳过而非整体报错，保证评测在不完美数据上也能跑。
- **R-4.1.4 规模可控**：支持限制样本数 / 每样本问题数，便于在可控成本下快速跑通。

### 4.2 灌入与召回

- **R-4.2.1 走真实写路径**：benchmark 对话经 `observe` 进入 memory 的真实抽取链路，不绕过抽取。
- **R-4.2.2 样本隔离**：不同样本的记忆互不串味（各自独立的记忆库）。
- **R-4.2.3 召回可观察**：用 benchmark 问题召回后，能看到召回了哪些记忆条目（含所属层、文本等）。

### 4.3 判分扩展点

- **R-4.3.1 稳定判分接口**：定义"给定问题 + 召回上下文 → 判分结果"的统一接口。
- **R-4.3.2 本期空实现**：本期提供不打分的占位实现（只展示召回供人看），未来接入 LLM-as-judge 时新增实现即可，不动编排 / 展示。

### 4.4 可用性与可观测

- **R-4.4.1 一键运行**：通过 `scripts/` 一键运行，开发者不需记忆底层命令（呼应 [`cross-platform-dev`](../../../.cursor/rules/cross-platform-dev.mdc)）。
- **R-4.4.2 真实调用前置告知**：运行会触发真实 LLM 抽取，机制需在文档 / 脚本提示中明确这一点（呼应 [`llm-api-confirm`](../../../.cursor/rules/llm-api-confirm.mdc)）。
- **R-4.4.3 结果可读展示**：召回结果以可读形式呈现（问题 / 标准答案 / 召回条目）。

## 5. 使用约束

- **数据集许可**：PerLTQA 为 **CC BY-NC 4.0（仅非商用研究）**。本机制仅用于**内部召回评测**，**不分发数据集、不将其用于商用模型训练**。基准数据文件**不纳入版本控制**（按 `.gitignore` 的 `data/` 规则忽略），由使用者自行下载。
- **真实 LLM 调用**：灌入阶段的记忆抽取会调用真实 LLM，按 [`llm-api-confirm`](../../../.cursor/rules/llm-api-confirm.mdc) 每次运行前需获授权；单测一律用 fake LLM，不触发真实调用。

## 6. 验收标准

> 本节 AC 全部是**机制层面的过程性标准**——验证"评测机制能落地、能跑通"，**不含任何 memory 召回质量 / 准确率指标**。

- **AC-1 加载跑通**：能加载至少一个 benchmark（PerLTQA / LoCoMo）的真实数据并归一成统一样本结构，限制样本 / 问题数生效。
- **AC-2 端到端跑通**：能把至少一个样本的对话经 `observe` 灌入 memory 真实写路径、再用问题走 `retrieve`，完整跑完并输出可读的召回展示（人工确认机制确实跑通、确有召回内容产生即可，不评判召回对不对）。
- **AC-3 可插拔验证**：至少两个 benchmark（中文 PerLTQA + 英文 LoCoMo）共用同一套适配 / 编排 / 展示，仅 loader 不同。
- **AC-4 判分扩展点就位**：判分接口存在且有空实现（NoopJudge），编排能调用它（即便不打分）。
- **AC-5 机制自测**：benchmark 解析与灌入/召回适配有单测覆盖，且单测用 fake LLM、不触发真实 LLM；`./scripts/check` 全绿。
- **AC-6 一键运行**：提供 `scripts/` 双端脚本，一条命令即可运行评测。

## 7. 开放问题 / 未来演进

> 以下不在本期验收范围内，记录以备后续推进：

- **Q-1 LLM-as-judge 量化判分**：接入后才能从"展示召回"升级为"可量化准确率"，是本机制最自然的下一步（对应 R-4.3 的扩展点）。
- **Q-2 抽取过度概括的短板**：首轮 PerLTQA 试跑观察到，memory 的 episodic 抽取偏高度概括，对"对话细节型"问题召回不足——这是 memory 侧（008）的潜在改进点，由本机制持续观察，但**修复属 008 范畴，不在 011**。
- **Q-3 更多 benchmark / 更多口径**：是否引入 LongMemEval、纯检索口径、向量召回对比等，按未来需要再评估。
- **Q-4 评测产物留档**：是否把每次评测结果结构化落盘 / 形成可对比的历史记录。
