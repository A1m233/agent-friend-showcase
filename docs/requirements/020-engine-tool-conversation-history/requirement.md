## 1. 背景与价值

### 1.1 现状

[005](../005-engine-tool-calling-and-web-search/requirement.md) 给 agent 装了 `web_search`，让它能"查外面"——不再活在训练数据截止时间里。

但 AI 朋友对**自己内部的"过去"**仍然是失忆的：

- **跨 session**：用户每隔几天聊一次（桌宠场景），AI 完全不记得上次聊了什么
- **同 session 长对话**：[009](../009-engine-context-management/requirement.md) 的上下文管理引入截断后，早期对话不在 prompt 里了
- **被动 vs 主动**：[008](../008-engine-memory/requirement.md) 的记忆模块负责"把最相关的记忆主动注入 prompt"——但 LLM 自己**没有**"我去翻一下过去聊过什么"的主动取回路径，类似真人"哎我们上次是不是聊过这个 / 你之前提过的那个朋友最近怎么样"

### 1.2 痛点

这与 [`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 第 1 条 **"像真人，不像工具"** 直接冲突——朋友不会忘记重要的对话；即使一时想不起，也会"翻翻聊天记录"再回应。

### 1.3 与 005 / 008 的关系

- **005**：建立工具调用机制 + `web_search`（外向取回）。本期是"内向取回"——同样走 Tool Protocol，是 005 抽象的第二个具体落地。
- **008**：被动记忆注入。两者互补，**不冲突**：memory 决定"什么自动浮现进 prompt"；本工具决定"AI 主动想去翻时翻得到"。LLM 在不同场景挑不同路径——记忆够用就不调工具，记不清就调。

### 1.4 为什么不顺手做更多工具

调研 Claude Code 工具池后，对陪伴型 agent 真正"撑得起独立需求"的暂仅本工具一项；其它有价值方向（WebFetch / 主动推送 / 提醒）见 §8 未来扩展，单独立项。

## 3. 非目标（Out of Scope）

以下能力**本期明确不做**，避免范围蔓延：

- **跨 user 隔离 / 多用户场景**——单设备单用户假设（与 005 一致）
- **结构化检索 / 分词 / 语义相似度**——本期 substring 模糊匹配，未来如果不够再升级
- **持久化索引 / 全文搜索引擎**——直接扫文件够用，YAGNI
- **可视化 UI**——本期纯 LLM 自助
- **删除 / 编辑会话内容**——只读
- **跨设备 / 跨账号同步**——data ownership 在本机
- **会话记录之外的"内向取回"**（如系统状态 / 自我配置查询）——各自单独立项

## 4. 核心需求详述

### 4.1 工具能力

**目标**：让 AI 朋友具备"主动翻看过去聊过的事"的能力。

需求点（用户/系统视角）：

- **R-4.1.1 关键字模糊匹配**：能用关键字检索过往对话内容（substring + 大小写无关，中英文均支持）。具体匹配算法由 `design.md` 决定。
- **R-4.1.2 时间范围过滤**：能限定 since / until 时间窗。时间参数**同时支持** ISO 8601 字符串和常见自然语言短语（如"3 天前"、"上周"等）。**解析逻辑必须抽成独立函数**，便于本期最简实现 + 未来扩展更多短语 / 升级解析能力。具体支持哪些短语、解析失败的兜底由 `design.md` 决定。
- **R-4.1.3 说话方过滤**：能限定只看用户说的 / 只看 agent 说的 / 不限。**对外参数语义不暴露 `user` / `assistant` 字眼**（采用 you / me 等拟人称谓），具体参数命名由 `design.md` 决定。
- **R-4.1.4 跨 session 默认开**：默认查询所有历史 session，**不暴露 session 是分桶概念**——LLM 不应感知"会话 id"。当前仍在进行的 session 也包括在内（以已写盘事件为准）。
- **R-4.1.5 limit 控制**：可限定返回条数；默认值与上限由 `design.md` 决定。

### 4.2 不暴露 AI 身份

**目标**：LLM 看到工具结果后**不应**脱口而出技术化表达，如"我查询了会话记录"、"在 session 数据中找到"、"event_type 为 user_message" 等。这是本工具最关键的边界。

需求点：

- **R-4.2.1 工具描述措辞克制**：description 用"回忆过去聊过的事"等拟人措辞，**不**直白说"查询数据库 / 历史日志"。
- **R-4.2.2 返回纯文本**：返回值是日记 / 叙事风格的纯文本，**不**含 JSON / schema 字眼（不出现 `session_id` / `event_type` / `payload` / `role` 等技术字段名）。
- **R-4.2.3 称谓拟人**：用户消息渲染为"你说"；assistant 消息渲染为"我说"；时间用人话格式（如"3 天前周一晚上 22:14"），**不**直接吐 ISO 字符串给 LLM。
- **R-4.2.4 inline reminder**：返回文本末尾追加引导（参考 005 `web_search` 的 `_format` 模式），提醒 LLM 用朋友口吻自然提及、不暴露查询动作。
- **R-4.2.5 空结果拟人**：找不到相关记录时，返回"我翻了翻好像没聊过"等拟人话术，**不**返回空列表 / "No results" 字样。

### 4.3 与既有机制对齐

- **R-4.3.1 走 Tool Protocol**：作为 `agent.tools.builtin.*` 的新成员实现 005 §4.1 的 Tool Protocol，由 `make_default_registry` 注入；**不改主调用循环、不引入新框架**。
- **R-4.3.2 调用历史可追溯**：与 005 R-4.1.3 一致——本工具的调用入参 / 结果进 `tool_call_request` / `tool_call_result` events，可重放。
- **R-4.3.3 失败拟人兜底**：参数解析错 / store IO 错时走 `ToolResult.is_error=True` 路径（参考 005 R-4.1.4 与 `web_search` 错误映射），LLM 拟人继续，对话不中断。
- **R-4.3.4 调试可观测沿用既有**：CLI 在 stderr 渲染 `[tool] <tool_name>(<args_brief>)` 时沿用 005 §4.8 的 `_brief` 截断，不特殊处理。

## 5. 关键体验原则

引用 [`0001 §1.3`](../../decisions/0001-product-vision-and-roadmap/README.md) 三条核心原则，本期具象化为：

1. **像真人，不像工具**
   - 朋友"想起过去"是自然行为，不该让人感觉到"AI 在调 API"
   - 措辞底线：不暴露查询动作、schema、ISO 时间戳等任何技术信息

2. **底层可替换，上层稳定**
   - 时间解析抽函数，本期最简实现 + 未来升级路径开放
   - 工具实现遵循 005 已建立的 Tool Protocol，不引入新框架

3. **失忆话术红线（在工具可用前提下进一步收紧）**
   - [001 R-4.2.5](../001-foundation-chat-and-memory/requirement.md) 已禁止"作为 AI / 训练数据截止"类话术
   - [005](../005-engine-tool-calling-and-web-search/requirement.md) 在 `web_search` 可用前提下进一步禁了"我无法访问实时信息"
   - 本期再进一步：本工具可用的前提下，"我不记得我们之前聊过什么 / 我无法访问历史对话"类话术**也**进入红线——既然能翻就不该说

## 6. 验收标准

- **AC-1 关键字检索可用**：用户问"我们上次聊过日语吗"（或类似回忆问句），AI 触发工具、找到结果、用朋友口吻自然提及，**不暴露 schema 字眼**（如 session_id / role / event_type）。
- **AC-2 没记忆时拟人兜底**：用户问的事真没聊过，AI 用"翻了翻好像没聊过"等拟人话术回应，不返回空列表 / 错误码 / "No results"。
- **AC-3 跨 session 合并查询**：在多个历史 session 各埋一条相关记录，触发查询能合并、按时间合理排序、按 limit 截断。
- **AC-4 时间范围过滤**：用户问"我们上周聊到那个事是什么来着"，AI 能用时间参数缩窄范围，结果限定在该窗口内；ISO 8601 与自然语言短语（至少包含"几天前 / 昨天 / 上周"中的一部分）都能用。
- **AC-5 工具调用可追溯**：工具调用入参 / 结果进 session events，正常退出 + `--resume` 后能看到历史调用链。
- **AC-6 失败拟人兜底**：模拟参数解析错 / store IO 错，AI 不崩溃 / 不暴露错误码，用拟人话术继续。
- **AC-7 失忆话术红线**：本工具可用的前提下，AI **不应**出现"我不记得我们之前聊过什么 / 无法访问历史对话"类话术。
- **AC-8 既有行为不退化**：001 ~ 019 已交付的所有行为继续可用，无回归。

## 7. 开放问题 / 待技术方案决策

> [`0002`](../../decisions/0002-incubation-tech-stack/README.md) 中已锁定的项目级技术栈不重复。
> 以下问题属于**本需求的实现策略**，**不在本需求文档中拍板**，将在同目录下的 `design.md` 中讨论与决策：

- **Q-1 工具命名 + description 措辞**：name 用 `recall_past_chats` / `remember_conversation` / `search_memory` / 其它？description 怎么写得像"想起来"而非"查表"？
- **Q-2 参数命名与 JSON Schema**：speaker 参数用 `who: "user"|"agent"` / `said_by: "you"|"me"` / 其它？时间字段名 `since` / `until` / `from_time` / `to_time`？JSON Schema 怎么布局？
- **Q-3 时间解析的实现范围**：本期支持哪些自然语言时间短语（"3 天前" / "昨天" / "上周" / "今天上午" / ...）？解析失败的兜底（落 `is_error` 还是默认全时段）？这个独立函数放哪、签名怎么定？
- **Q-4 跨 session 扫描效率**：直接扫所有 session 文件每次重复读，还是用 store list + 按 mtime 倒序 lazy load？是否需要按时间窗预先 prune？是否需要防超大返回（截 result text 总长度）？
- **Q-5 渲染格式细节**：每条记录展示几行（匹配那条 + 上下文配对一起展示，还是只展示匹配那条）？时间字符串具体格式？多条之间怎么分隔？
- **Q-6 inline reminder 文本**：参考 005 `web_search` 的 `_format` 末尾，写到什么程度？
- **Q-7 limit 的默认值 / 单次返回 token 预算**：5 / 10 / 20 / 其它？返回 token 大致控在多少（避免单次 tool result 把上下文吃满）？
- **Q-8 与 008 memory / 004 system prompt 的运行时关系**：是否要在 system prompt 里加一句"如果记不清就用本工具"？还是依赖 description 让 LLM 自主判断？
- **Q-9 `make_default_registry` 的依赖注入**：当前 factory 无参；本工具需要 `SessionStore` 实例，factory 签名怎么改最小破坏？

## 8. 未来扩展（已识别 / 本期不做）

调研 Claude Code 工具池时识别但本期不做的方向，作为后续需求线索：

- **WebFetch（按 URL 取网页内容）**：用户分享链接时"看看这页讲什么"。与 005 `web_search` 同类（外向取回），nice-to-have；可在下一轮工具扩展中顺手做。
- **主动推送 / 定时提醒（cron-like）**：朋友帮你"明天 9 点提醒开会"——典型陪伴场景。但 agent-friend 当前是"用户发→AI 回"被动循环，做主动推送涉及桌宠层 / bridge 调度改动，**架构跨度大，应作为独立需求立项**（非本期 Tool Protocol 内可解决）。
- **结构化检索 / 语义相似度**：本期 substring 简单实现，未来如果用户反馈"找不全 / 找不准"再升级。
- **更多 Claude Code 借鉴**：编程类工具（Read / Bash / Grep / LSP / Worktree / Notebook / ...）与陪伴 agent 不匹配；对话型工具（AskUserQuestion）本身就在和用户对话，无意义。**本工具池基本已榨干**——后续工具应来自陪伴场景的具体需求，而非"还能补什么"。
