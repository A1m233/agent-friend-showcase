# 035 · 编辑并重发最后一条用户输入

> Edit and Resend Latest User Query

让用户在对话中发现最近一次输入写错时，可以直接编辑这条 user query 并重新发送。行为上等同于回到这条输入之前，用改后的内容重新跑一轮对话。

## 1. 背景与价值

### 1.1 现状

当前 chat 窗已经支持：

- 底部 `ChatSender` 输入并发送消息。
- 用户/助手消息列表展示。
- 用户消息 action bar 里有复制能力。
- 历史会话切换后可继续对话。

但用户如果刚发出一条内容后发现输错，只能继续发一条补充说明，或者重新开一段会话。对于错别字、漏掉上下文、表达不清这类高频场景，体验不自然。

### 1.2 这次要做什么

本期增加“编辑并重发最后一条用户输入”能力：

- 用户可以点击最近一条可编辑 user 消息上的编辑按钮。
- 消息位置出现一个输入区域，初始内容是原 user query。
- 输入区域右下角提供“取消”和“发送”操作。
- 发送后，当前对话的活跃视图回到这条 user query 之前，并用修改后的 query 重新发起一轮对话。

这个需求只解决“刚刚发错内容想重输”的场景，不做完整的任意历史分支编辑系统。

## 2. 范围

### 2.1 包含

- **R-1 编辑入口**：仅最近一条可编辑 user 消息显示编辑入口。入口应与现有复制、时间等 action bar 行为一致，不影响 assistant 消息。
- **R-2 内联编辑输入区**：点击编辑后，该 user 消息位置切换为内联输入区，内容预填原 query。输入区右下角有取消和发送按钮。
- **R-3 Sender 封装复用**：对 `@tdesign-react/chat` 的 `ChatSender` 做一层项目级封装，通过参数区分底部输入态和编辑输入态。底部 composer 与内联编辑区统一使用该封装，为后续附件等共同能力留口子。
- **R-4 取消编辑**：点击取消后退出编辑态，原消息与原对话视图不变，不产生网络请求或会话变更。
- **R-5 编辑后重发**：点击发送后，以编辑后的文本重新发起对话。当前活跃视图中旧的错误 user query 及其后续 assistant 回复不再作为后续对话上下文继续展示或续写。
- **R-6 状态互斥**：流式回复中、历史会话加载中、语音通话阻塞文字输入时，不允许进入或提交编辑重发。一次只允许编辑一条消息。
- **R-7 输入校验**：编辑内容去首尾空白后为空时不可发送。发送成功前应避免重复提交。
- **R-8 历史一致性**：重发完成后，历史会话列表、当前会话恢复指针、刷新后重新打开的会话视图，应与当前活跃视图语义一致。

### 2.2 不包含

- 任意历史 user query 的编辑和回溯。
- assistant 消息编辑、重新生成同一条 assistant 回复、手动选择多个分支。
- 分支会话管理 UI，例如查看旧分支、切换分支、命名分支、删除分支。
- 对已经产生的工具外部副作用做撤销。
- 语音输入、附件上传、persona/model 切换 UI 等输入框扩展功能本身。
- 长期记忆的完整可视化修复工具。本期只要求设计上避免旧错误 query 静默污染后续活跃对话。

## 3. 核心需求详述

### 3.1 可编辑消息定义

本期“可编辑”只指当前会话中最后一个 user turn：

- 当前没有 streaming。
- 当前没有 history loading。
- 这条 user 消息之后没有新的 user 消息。
- 当前文字输入未被语音通话等全局状态禁用。

如果当前会话最后一轮 assistant 处于 error 状态，只要不在 streaming 中，仍允许编辑最后一条 user query 后重发。

### 3.2 内联编辑体验

用户点击编辑按钮后：

- 原 user 气泡位置变为内联输入区。
- 输入区预填原 query，用户可以直接修改。
- 操作区位于输入区右下角，包含取消和发送。
- 取消恢复原消息展示。
- 发送进入提交状态，避免二次点击。

内联编辑区应与底部输入框保持一致的基础输入行为，例如输入事件同步、IME Enter 行为、禁用态风格和发送按钮可用性。底部 composer 的语音输入等业务能力不要求出现在编辑区。

### 3.3 重发后的对话语义

编辑并发送后，用户视角应看到：

- 错误的原 user query 不再作为当前活跃对话的一部分出现。
- 原 user query 后面的旧 assistant 回复不再出现在当前活跃视图中。
- 新的 user query 出现在原位置或等价的最新轮位置，并触发新的 assistant 流式回复。
- 后续继续发送消息时，agent 以上述新 query 和新回复作为上下文延续。

底层是否保留旧事件用于审计、如何表达“旧分支已被回溯”、是否创建新 session，属于 `design.md` 的技术方案问题；需求侧只要求用户看到和续聊的活跃语义正确。

### 3.4 Sender 封装边界

新增封装的目标是降低重复和统一输入行为：

- 封装组件接收参数区分 `bottom` 与 `edit` 位置。
- 通用输入能力放在封装里，位置特有的业务能力由外层组合。
- 第一版允许保持轻量透传，不提前实现附件系统。
- 后续附件、统一快捷键、输入区扩展等共同能力，应优先在该封装内演进，而不是分别改底部 composer 和编辑区。

## 4. 验收标准

- **AC-1 编辑入口限制**：只有当前会话最后一条符合条件的 user 消息显示编辑按钮；非最后一条 user 消息和 assistant 消息不显示编辑按钮。
- **AC-2 编辑 UI**：点击编辑后，消息位置出现预填原文的输入区，右下角有取消和发送按钮。
- **AC-3 取消无副作用**：修改内容后点击取消，恢复原消息展示，不触发新请求，不改变当前会话。
- **AC-4 空内容不可发送**：编辑内容为空或全空白时，发送不可用或提交无效。
- **AC-5 重发成功**：修改最后一条 user query 后发送，旧 query 和旧 assistant 回复不再出现在当前活跃视图中，新 query 触发新的 assistant 流式回复。
- **AC-6 续聊上下文正确**：重发完成后继续发送下一条消息，agent 不应基于旧错误 query 的 assistant 回复继续答复。
- **AC-7 状态互斥**：streaming、history loading、文字输入被语音通话阻塞时，不能进入或提交编辑重发。
- **AC-8 刷新/重开一致**：重发后刷新历史列表或重新打开该会话，当前活跃视图仍与重发后的视图一致。
- **AC-9 Sender 复用**：底部 composer 和编辑输入区均使用项目级 Sender 封装；实现中不出现两套分叉的 ChatSender 基础行为。

## 5. 开放问题 / 待 design.md 决策

- **Q-1 回溯持久化语义**：同一 session 内追加回溯 marker，还是创建新 session/fork？如何保证刷新后 active projection 一致？
- **Q-2 事件边界**：最后一条 user turn 的后续 assistant/tool/compaction/system marker 如何从活跃上下文中排除？
- **Q-3 memory 影响**：旧错误 query 及旧 assistant 回复如果已进入长期记忆，如何避免污染后续活跃对话？
- **Q-4 前端状态模型**：编辑态放在 MessageList 局部 state、conversation store，还是独立 UI store？
- **Q-5 Sender 封装 API**：位置参数、slots、ref、IME guard、native event 兜底如何组织，既复用又不把底部 composer 的语音业务耦合进编辑区？
- **Q-6 失败恢复**：重发请求失败、后端回溯成功但新回复失败、用户中途停止流式时，UI 应如何展示和恢复？
