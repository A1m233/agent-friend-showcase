# 需求记录（Requirements）

本目录存放 `agent-friend` 的**需求文档**及其配套的**技术方案文档**。

## 适用范围

- 一个需求 = 一个独立交付的功能 / 能力 / 模块
- 每个需求的"做什么"和"怎么做"分别由两份独立文档承载

> 影响多个需求的项目级决策不属于这里，应放在 `docs/decisions/`。

---

## 目录约定

每个需求都是一个**独立文件夹**，命名格式为 `<3 位数字编号>-<kebab-case 英文短语>`：

```
docs/requirements/
├── README.md                              ← 本文件，仅维护管理规则
├── 001-foundation-chat-and-memory/
│   ├── requirement.md                     ← 需求文档（"做什么"）
│   ├── design.md                          ← 技术方案（"怎么做"，按需创建）
│   └── （未来可放图、tasks.md、test-plan.md、assets/ 等）
└── 010-desktop-pet/
    ├── requirement.md
    └── ...
```

约定要点：

- **编号 3 位数字、连续递增**：方便排序，3 位预留扩展空间。
- **kebab-case 英文短语**：跨平台路径友好。
- **一个需求一个文件夹**：未来可在同一需求下放图、tasks.md、test-plan.md、assets 等。
- **编号不必紧密连续**：可以为不同主题预留区段（如 `001-009` 留给基础能力、`010-019` 留给形态层、`020-029` 留给感官通道等），不强制。

---

## 文件命名约定

| 文件               | 必需性 | 职责                                                                 |
| ------------------ | ------ | -------------------------------------------------------------------- |
| `requirement.md`   | 必需   | 描述这个需求要**做什么**：背景、范围、需求点、验收标准、开放问题     |
| `design.md`        | 按需   | 描述这个需求要**怎么做**：模块划分、数据流、接口设计、关键算法选型 |
| `tasks.md`         | 可选   | 拆分到可执行的任务清单                                               |
| `test-plan.md`     | 可选   | 测试用例与策略                                                       |
| `assets/`          | 可选   | 配图、流程图源文件、原型截图等                                       |

> 不强制每个需求都有 `design.md`——如果某个需求小到不需要单独的技术方案，可以省略。但凡涉及非平凡技术决策的，都应当有。

---

## 需求文档（requirement.md） vs 技术方案（design.md）的边界

| 维度            | requirement.md                            | design.md                                       |
| --------------- | ----------------------------------------- | ----------------------------------------------- |
| 回答的问题      | 做什么、为什么做、做到什么程度算成功      | 怎么做、模块怎么划、接口怎么定、算法怎么选     |
| 受众            | 任何想了解这个需求的人（含产品/业务视角） | 实现者（开发者）                                |
| 变更频率        | 较低，需求确定后不应频繁变                | 较高，实现细节会随调研深入而调整               |
| 是否含技术选型  | 不含具体技术选型                          | 含具体实现的算法 / 库 / 数据结构选型            |
| 是否含验收标准  | 含                                        | 不含（验收标准属于需求侧）                      |

简单原则：

> **如果一段内容讲的是"用户/系统应该看到什么效果" → 进 `requirement.md`**
>
> **如果一段内容讲的是"为了达到那个效果，代码怎么写" → 进 `design.md`**

---

## 与决策（decisions）的边界

需求文档**不重复**项目级技术栈决策。

- 项目级技术问题（用 Python 还是 TS、用哪个 LLM Provider、用什么存储方案、目录结构怎么定）→ 已在 `docs/decisions/` 中锁定，需求文档**直接指向 decisions** 即可，不复述、不再讨论。
- 仅服务本需求的实现选择（本期记忆抽取用规则 / LLM、本期 prompt 怎么写、本期上下文截断策略）→ 写在本需求的 `design.md`。

如果在写 `requirement.md` 时遇到"这个看起来是项目级问题"的情况，应**先评估是否要新建一个决策**（在 `docs/decisions/` 下），而不是把它塞进当前需求文档。

---

## 修改约定

- **需求是会演进的**：与决策不同，需求允许就地修订（业务理解会深化、范围会调整）。
- **重大范围调整应留痕**：如果调整大到改变了原本的"在做什么"，应在文档元信息里标注修订时间与简短原因，方便下游 `design.md` 等文档同步评估影响。
- **下游影响**：修改 `requirement.md` 时，必须评估对应 `design.md`（如已存在）是否需要同步调整。
